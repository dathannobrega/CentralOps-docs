Versão: 1.7 (legada)

Runbook: SLO em risco (latência alta)

Latência de coleta-para-normalização excedem SLA. Alertas chegam ao Wazuh > 5 minutos atrasados.

Para: SRE.

SLA targets​

MétricaTargetAlertar se
Coleta → Fila< 1 min> 2 min
Normalização< 30 seg> 1 min
Despacho< 1 min> 2 min
E2E (Coleta → Wazuh)< 3 min> 5 min

"E2E" é o que importa — desde evento é criado no vendor até chegar em Wazuh Indexer.

Diagnóstico rápido​

1. Verificar SLO atual​

Vá a Pipeline Health → procure por gráfico "Latency (seconds)".

Se vê: 180s (3 min) ou menos → OK
Se vê: 300s+ (5 min) → SLO em risco

Ou via métrica Prometheus (exposta pelo OTel Collector em :8889, não por worker):

curl http://localhost:8889/metrics | grep collector_
# dentro do compose: curl http://otel-collector:8889/metrics | grep collector_

2. Qual integração está lenta?​

Pipeline Health → card por integração.

Procure por "Latency" alta:

Sophos: 1 min (OK)
Defender: 2 min (OK)
NinjaOne: 8 min (LENTO!)

Conclusão: NinjaOne está causando SLO burn.

3. Qual etapa está lenta?​

Opções:

A) Coleta lenta (vendor API lenta ou taxa limite):

Sintoma: última coleta demorou 2 min
Ação: vá para "Causa 1" abaixo

B) Normalização lenta (mapping complexo):

Sintoma: evento coletado, mas demora 1 min para normalizar
Ação: vá para "Causa 2"

C) Despacho lento (dispatcher sobrecarregado):

Sintoma: evento normalizado, mas demora 2 min para chegar ao Wazuh
Ação: vá para "Causa 3"

Causas e soluções​

Causa 1: Coleta lenta​

Sintomas:

  • Vendor API respondendo lento (> 30s por request).
  • Muitos eventos por coleta (vendor retorna 10k eventos, parse demorado).
  • Taxa limite (429 retries).

Diagnóstico:

docker compose logs collector-worker-priority | grep NinjaOne | tail -20
# Procure por:
# - "API response time: 45s" → API lenta
# - "429 Too Many Requests" → taxa limite
# - "Collected 10000 events" → volume alto

Solução:

  1. Se API lenta: aumentar intervalo de coleta (ex: 5 min → 10 min). Trade-off: dados mais antigos.

    Vá a Administração → Integrações → NinjaOne
    Editar → alterar "Collection interval" (se disponível)

  2. Se taxa limite: diminuir throttle no backfill ou reduzir parallelismo.

    redis-cli HSET "collection:config:ninjao" throttle 5
    # Próxima coleta usa throttle=5 (mais lento, menos requests/s)

  3. Se volume alto: avisar ao cliente que há muitos eventos. Filtrar no vendor (se possível).

Causa 2: Normalização lenta​

Sintomas:

  • Mapping DSL com loops/regex complexos.
  • Muitas regras condicionais (if/then/else profundamente aninhados).

Diagnóstico:

# Executar mapping em 1 evento teste, medir tempo
# (ferramentas: curl + time)

curl -X POST http://localhost:8000/api/test-mapping \
  -d '{"platform": "ninjaone", "raw_event": {...}}'

# Se responde > 1s: mapping é lento

Solução:

  1. Simplificar mapping: remover regras não-essenciais.

    Exemplo: múltiplas if/then aninhadas → consolidar em 1 regra

  2. Usar defaults: campos que não precisam transformação, usar padrão.

    Antes: severity = if ... else if ... else if ... else "low"
    Depois: severity = if vendor_critical then "critical" else "low"

  3. Cache: para transformações repetidas (roadmap).

Causa 3: Despacho lento​

Sintomas:

  • Dispatcher queue entupida (depth > 1000).
  • Dispatcher latência > 2s por evento.
  • Wazuh Indexer respondendo lentamente (yellow/red status).

Diagnóstico:

# Fila de despacho
redis-cli LLEN dispatch
# Se > 1000: fila cresce

# Dispatcher memory/CPU
docker stats collector-dispatcher

# Wazuh Indexer status
curl -k -u admin:admin https://wazuh-indexer:9200/_cluster/health | jq '.status'
# Se "yellow" ou "red": Wazuh tem problema

Solução:

  1. Escale dispatcher:

    docker compose up -d --scale collector-dispatcher=5
    # Aguarde 5 min, verificar fila

  2. Escale Wazuh Indexer (se status yellow/red):

    Aumentar replicas, aumentar heap memory, limpar índices antigos
    → Contatar Wazuh admin

  3. Batch tunning: reduzir tamanho de batch (envia mais frequente, menos latência pico).

Recovery: burst de latência​

Se latência subiu subitamente (foi 1 min, agora 8 min):

Passo 1: Scale up imediatamente​

# Aumentar workers coletores
docker compose up -d --scale collector-worker-priority=10

# Aumentar dispatchers
docker compose up -d --scale collector-dispatcher=5

# Aguardar ~2 min

Passo 2: Investigar causa​

Enquanto workers escalam, investigar:

# Vê em logs?
docker compose logs --tail=100 | grep -i "error\|429\|timeout"

# Vendor mudou algo?
# Novo cliente com volume alto?
# Wazuh offline?

Passo 3: Long-term fix​

Após SLO normalizar:

  • Aumentar workers permanentemente.
  • Otimizar mapping ou coleta.
  • Conversar com vendor sobre rate limit.
  • Escalar Wazuh se necessário.

Prevention​

  1. Monitor continuous: dashboard com latency trend (últimas 24h).
  2. Alerting: PagerDuty/Slack se latency > 5 min × 10 min.
  3. Capacity planning: se crescimento de 10% × ano, previamente escale.
  4. Load testing: teste nova integração em staging antes de produção.

SLA Reporting​

Para compliance/stakeholders, coletar:

Period: 2026-04-01 to 2026-04-30
E2E Latency:
  - P50: 1.2 min (median)
  - P95: 2.8 min (95th percentile)
  - P99: 4.5 min
  - Max: 8.2 min (incident day)

Availability: 99.8% (downtime < 30 min)

Incidents: 1 (NinjaOne API slow on 2026-04-15)

  • Monitorar latência? Vá a Pipeline Health.
  • Escalar workers? Vá a Collectors.
  • Otimizar mapping? Vá a Mappings.
