Versão: 1.7 (legada)

Runbook: Scheduler preso (RedBeat)

Integração está em "Ativo" mas não coleta eventos (última coleta > 15 min atrás).

Para: SRE, Engineer.

Diagnóstico rápido​

1. Beat está rodando?​

docker compose ps collector-beat
# Status deve ser "running"

Se parado:

docker compose restart collector-beat
# Esperar 10s, verificar:
docker compose logs collector-beat | tail -10

2. Schedule existe no Redis?​

redis-cli KEYS "redbeat::schedule:*"
# Deve listar entradas tipo "redbeat::schedule:sophos-alerts"

# Se lista vazia: schedule foi deletado, ver seção "Schedule deletado"

3. Integração está ativa?​

Vá a Administração → Integrações → clique integração.

  • Estado: deve ser "active" (não "paused", "error").
  • Última coleta: deve ser recente (< 5 min).

Se estado = "error": há erro de conexão (credenciais expiradas?). Veja Integrações.

Ações por cenário​

Cenário A: Beat parou​

Sintoma: docker compose ps mostra collector-beat como "exited".

Ação:

docker compose restart collector-beat

# Aguardar ~5s
docker compose logs collector-beat | grep -i "start\|schedule"
# Procure por "Scheduler: Loading entries from Redis"

Se continua falhando:

docker compose logs collector-beat | grep -i "error\|exception"
# Vê qual erro?

Causa comum: Redis inválido ou desligado.

redis-cli ping
# Se não responde: redis parou
docker compose restart redis

Cenário B: Schedule deletado​

Sintoma: redis-cli KEYS "redbeat::schedule:*" vazio OU integração não aparece.

Ação:

# Deletar e recriar integração
# Vá a Administração → Integrações → clique integração → Deletar

# Recriar: + Adicionar integração → preencha dados → Salvar

# Beat auto-detecta nova integração (~5s)
redis-cli KEYS "redbeat::schedule:*"
# Deve listar a nova entrada

Cenário C: Workers não pegam tasks​

Sintoma: Beat está ok, schedule existe, mas tasks não executam.

Ação:

# 1. Workers estão rodando?
docker compose ps | grep collector-worker
# Deve haver N replicas "running"

# 2. Se parados:
docker compose up -d collector-worker-priority

# 3. Se rodando mas task não executa:
docker compose logs collector-worker-priority | tail -20
# Procure por motivo (permissão, import error, etc)

Cenário D: Fila entupida​

Sintoma: schedule dispara, mas task fica na fila (não executa).

Ação:

# 1. Verificar tamanho das filas
redis-cli LLEN collect.priority
redis-cli LLEN collect.bulk
# Se > 1000: fila entupida

# 2. Escalar workers
docker compose up -d --scale collector-worker-priority=5

# 3. Aguardar ~5 min, verificar tamanho fila novamente

Cenário E: Credenciais expiradas​

Sintoma: integração em estado "error", mensagem "401 Unauthorized".

Ação:

Vá a Administração → Integrações → clique integração.

  1. Clique Editar credenciais.
  2. Preencha novas credenciais (ex: novo Client Secret em Sophos).
  3. Clique Testar conexão. Se OK:
  4. Clique Salvar.

Estado volta a "active", coleta retoma em < 1 min.

Diagnóstico avançado​

Inspecionar schedule entry​

# Ver uma entry específica
redis-cli HGETALL "redbeat::schedule:sophos-alerts"

# Saída esperada:
# 1) "next_run_time"
# 2) "2026-04-27T10:05:30+00:00"
# 3) "task"
# 4) "collectors.collect_vendor_logs_priority"
# 5) "kwargs"
# 6) "{\"integration_id\": 42}"

Se next_run_time é no passado (> 1 min atrás), Beat deveria ter disparado.

Cause: Beat travado ou lento.

# Verificar logs do Beat
docker compose logs collector-beat | tail -100 | grep -i "dispatch\|send"

# Se não vê "Sending due task", Beat está travado
# Restart:
docker compose restart collector-beat

Verificar task na fila​

# Listar tamanho da fila
redis-cli LLEN collect.priority

# Ver primeiras 5 tasks
redis-cli LRANGE collect.priority 0 4

# Se vazio: task foi executada (ou perdida)

Verificar logs do worker​

# Ver execuções recentes
docker compose logs collector-worker-priority | grep -i "completed\|success"

# Se vê "task completed": worker está processando
# Se não vê: worker está parado ou não recebendo tasks

Recovery procedures​

Recovery 1: Reset do schedule​

Se schedule corrompido ou travado:

# Deletar todas as entries (cuidado!)
redis-cli KEYS "redbeat::schedule:*" | xargs redis-cli DEL

# Beat será reiniciado, detectará integrations sem schedule
# Re-criará entries automaticamente (~5s)

# Verificar:
redis-cli KEYS "redbeat::schedule:*" | wc -l
# Deve match número de integrações ativas

Recovery 2: Rebuild Redis​

Se Redis está corrompido (offline, dados inconsistentes):

# Backup (se quiser preservar)
docker compose exec redis redis-cli BGSAVE

# Limpar Redis
docker compose exec redis redis-cli FLUSHDB

# Reiniciar Beat (detecta Redis vazio, re-cria)
docker compose restart collector-beat

# Aguardar ~10s
redis-cli KEYS "redbeat::schedule:*" | wc -l
# Deve listar integrações

Recovery 3: Rebuild workers​

Se workers estão zumbis (rodando mas não processando):

docker compose restart collector-worker-priority collector-worker-bulk

# Aguardar ~5s
docker compose logs collector-worker-priority | grep -i "started\|ready"

Prevention​

  1. Monitoring: alertar se "última coleta > 10 min para integração ativa".
  2. Healthchecks: Beat + workers têm healthchecks docker (restart automático se falham).
  3. Backup Redis: ativar RDB/AOF para durabilidade de schedule.

Próximos passos​

  • Monitorar saúde? Vá a Pipeline Health.
  • Alterar frequência de coleta? Vá a Scheduler.
  • Integração com problema? Vá a Integrações.
