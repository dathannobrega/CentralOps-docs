Versão: 1.7 (legada)

Runbook: Taxa alta de quarentena

Subitamente, 100+ eventos por hora aparecem em quarentena (antes era 0-5/h).

Para: SRE, Engineer.

Diagnóstico rápido​

Sintoma​

Vá a Pipeline Health → card "Quarantine" mostra vermelho (100+).

Ou: Dashboard → card "Quarantine" crescendo em tempo real.

1. O que mudou recentemente?​

Pergunte-se:

  • Vendor mudou API (resposta diferente)?
  • Mapping foi editado (pode ter quebrado algo)?
  • Integração nova criada?
  • Alta de tráfego (novo cliente, novo attack)?

2. Qual vendor está enviando erros?​

Vá a Operação → Quarentena.

Filtro: nada (ver todos).

Coluna "Platform" → quais vendors aparecem mais?

Exemplo resultado:
- Sophos: 200 (80%)
- Defender: 40 (16%)
- NinjaOne: 10 (4%)

Conclusão: Sophos está enviando errors

3. Qual tipo de erro?​

Vá a Operação → Quarentena → clique em 1-2 eventos Sophos.

Veja "Error message":

Opção A: "Required field 'timestamp' missing"
  → Sophos parou de enviar timestamp (schema change)

Opção B: "Type error: severity expected string, got int"
  → Sophos mudou tipo de field (era string, agora int)

Opção C: "JSON parse error"
  → Sophos retorna JSON inválido (bug ou encoding)

Ações por tipo de erro​

Tipo A: Campo obrigatório faltando​

Causa: Vendor removeu ou renomeou campo.

Exemplo: Sophos parou de enviar timestamp, começou a enviar event_timestamp.

Ação:

  1. Vá a Mappings → DSL do vendor (ex: sophos.alert).
  2. Localize regra timestamp = ....
  3. Mude para: timestamp = event_timestamp (ou field certo).
  4. Salve (cria v2).
  5. Volte a Quarentena → "Reprocess All Sophos".
  6. Aguarde 1 min. Erros devem sumir.

Validação: Quarentena volta a 0 ou muito baixo.

Tipo B: Mudança de tipo​

Causa: Vendor mudou tipo de field (int → string, vice-versa).

Exemplo: Sophos antes enviava severity: 5 (int), agora envia severity: "high" (string).

Ação:

  1. Vá a Mappings → DSL do vendor.
  2. Localize regra de severity.
  3. Adapte type casting: 
    Antes: severity = severity
    Depois: severity = string(severity)  // force string
    Ou: severity = if typeof(severity) == "int" then ... else ...
  4. Salve (v2).
  5. Quarentena → "Reprocess All".

Tipo C: JSON parse error​

Causa: Vendor retorna JSON malformado (rare).

Ação:

  1. Vá a Quarentena → expanda 1 evento com erro.
  2. Procure raw payload → é JSON válido? Procure por quote/escape errado.
  3. Se malformado: Discard All (eventos corrompidos, irreparáveis).
  4. Contate suporte do vendor: "JSON retornado está inválido".

Root cause investigation​

Hipótese 1: Vendor mudou API​

Teste:

# Chamar API do vendor diretamente
curl -H "Authorization: Bearer $TOKEN" \
  https://api.sophos.com/common/v1/alerts | jq '.alerts[0]' | head -20

# Comparar com mapping esperado
# Faltam fields? Novos fields? Tipos mudaram?

Se confirmado:

Contate vendor: "Documentação desatualizada, API schema mudou em [data]".

Atualize mapping conforme nova schema.

Hipótese 2: Mapping foi acidentalmente quebrado​

Teste:

Vá a Mappings → Histórico de versões da integração.

Compara v1 (anterior) com v2 (atual):

V1: severity = map(alert_level, {1: "low", 5: "critical"})
V2: severity = alert_level  // oops, removeu map!

Resultado: severity agora é int (5), não string ("critical")
Validação falha: expected string

Se confirmado:

Clique Rollback to v1. Status volta a "ativo".

Quarentena limpa em < 1 min.

Hipótese 3: Novo cliente/grande volume​

Teste:

Vá a Operação → Alertas.

Filtro: últimas 24h, plataforma Sophos.

Contagem antes vs. agora:

Antes: 1000 eventos/dia
Agora: 10000 eventos/dia

Taxa 10x. Mas qual % em quarentena?

Antes: 1% = 10 eventos
Agora: 3% = 300 eventos

Conclusão: taxa de erro subiu de 1% para 3%

Se confirmado: problema não é schema (que seria 100%), mas sim qualidade dos dados.

Possibilidades:

  • Novo cliente enviando dados "sujos" (muitos campos inválidos).
  • Attack em progresso (dados anormais).
  • Teste de carga (vendor testando).

Ação: investigar qual cliente / qual tipo de evento está falhando.

Bulk remediation​

Se 1000+ eventos em quarentena:

Opção A: Discard All (rápido)​

Vá a Quarentena
Filtro: Platform = Sophos
Clique: "Actions" → "Discard All"

Trade-off: eventos perdidos (não recuperáveis, mas logs ficarão em Wazuh/vendor).

Use se: eventos são noise ou teste.

Opção B: Fix + Reprocess (melhor)​

1. Identifique erro
2. Edite mapping (fix)
3. Quarentena → "Reprocess All"
4. Aguarde 5 min (reprocessamento em background)
5. Quarentena volta a 0

Use se: erro é mapeável (tipo, nome field, etc).

Opção C: Backfill + Bulk reprocess​

Se erro afeta muitos eventos históricos:

1. Crie backfill da integração (últimos 7 dias)
2. Backfill executa com novo mapping (v2)
3. Eventos novos + históricos normalizados com v2
4. Quarentena limpa

Prevention​

  1. Monitor dashboard: alerta se quarentena > 50 em 1h.
  2. Version history: sempre guarde mapping anterior (1-click rollback).
  3. Test integration: após major vendor update, testar antes de produção.
  4. Documentation: rastrear mudanças de vendor (changelog).

Próximos passos​

  • Editar mapping? Vá a Mappings.
  • Reverter mapping? Vá a Mappings > Versioning.
  • Triaging eventos? Vá a Quarentena.
  • Vendor mudou API? Vá a Integrações.
