Versão: 1.7 (legada)

Runbook: Dispatcher (Envio para Wazuh)

Sintomas, diagnóstico e ações para quando eventos não chegam ao Wazuh.

Para: SRE, Engineer.

Sintoma 1: "Nenhum evento em Wazuh Indexer"​

Diagnóstico​

1.1 Eventos estão sendo coletados?

Vá a Operação → Dashboard → card "Últimos eventos normalizados".

  • Sim (vê eventos): prossiga 1.2.
  • Não (vazio): problema é coleta, não dispatcher. Veja Collectors.

1.2 Dispatcher está rodando?

docker compose ps | grep dispatcher
# Deve estar "running"

docker compose logs dispatcher | tail -20
# Procure por erros

1.3 Status do dispatcher

Vá a Pipeline Health → card "Dispatcher".

  • Verde: conectado ao Wazuh, enviando.
  • Amarelo: conectado mas latência alta.
  • Vermelho: desconectado ou muitos erros.

Ações​

Se vermelho (desconectado):​

Causa possível: Wazuh Manager offline, porta 6514 bloqueada, TLS inválido

Checklist:

# 1. Wazuh está rodando?
curl -k https://wazuh-manager:55000/api/version
# Se erro: Wazuh offline

# 2. Porta 6514 acessível?
nc -zv wazuh-manager 6514
# Se erro: firewall bloqueando

# 3. Certificado TLS válido?
docker compose logs dispatcher | grep -i "ssl\|certificate"
# Se erro: cert inválido ou vencido

Ação corretiva:

# Reiniciar Wazuh Manager
docker compose restart wazuh-manager

# Esperar ~1 min, verificar:
docker compose logs dispatcher | grep "Connected\|Syslog"

Se amarelo (latência alta):​

Causa: muitos eventos, Wazuh Indexer lento

Ação:

Escale dispatcher:

docker compose up -d --scale collector-dispatcher=3

Teste manual​

# Enviar evento teste via Syslog
echo "TEST: event" | nc -w 1 wazuh-manager 6514

# Verificar no Wazuh se chegou
curl -k -u admin:admin https://wazuh-indexer:9200/wazuh-alerts-*/_search?q=TEST

Sintoma 2: "Eventos em quarentena com erro de despacho"​

Diagnóstico​

Vá a Operação → Quarentena.

Procure por erro tipo: "Dispatcher: Failed to send to Wazuh".

Ações​

Causa: evento tem schema inválido (ex: field obrigatório missing).

Solução: editar mapping, corrigir schema, reprocess

Vá a Quarentena para workflow completo.

Sintoma 3: "Eventos chegando ao Wazuh sem namespace _centralops"​

Diagnóstico​

# Verificar no Wazuh
curl -k -u admin:admin https://wazuh-indexer:9200/wazuh-alerts-*/_search?size=1 | jq '.hits.hits[0]._source'

# Procurar por "_centralops"
# Se não existe: enriquecimento não foi aplicado

Ação​

Dispatcher tem bug. Reiniciar:

docker compose restart collector-dispatcher

# Verificar logs pós-restart
docker compose logs dispatcher | grep "Enrich\|_centralops"

Sintoma 4: "Taxa de despacho lenta"​

Vou à Pipeline Health → "Dispatch rate" = 1 evento/min (esperado: 50+).

Diagnóstico​

# Fila de despacho entupida?
docker compose exec redis redis-cli LLEN dispatch
# Se > 1000: fila crescendo

# Dispatcher está processando?
docker compose logs dispatcher | grep "Sending\|batch"

Ações​

Se fila entupida:

# Escale dispatcher
docker compose up -d --scale collector-dispatcher=5

# Aguarde 5 min, verificar:
docker compose logs dispatcher | tail -20

Se Wazuh está lento:

# Verificar Wazuh Indexer health
curl -k -u admin:admin https://wazuh-indexer:9200/_cluster/health

# Se status = "yellow" ou "red": problema em Wazuh
# → Escalar para Wazuh admin

Symptom 5: "Dispatcher crash/restart contínuo"​

Diagnóstico​

docker compose logs dispatcher | grep -i "exception\|error\|crash"

# Se muito memory: memory leak
docker stats collector-dispatcher

Ações​

Se memory leak:

# Reiniciar dispatcher
docker compose restart collector-dispatcher

# Se problema persiste: possível bug em SDK Syslog
# → Abrir issue no GitHub

Se exception:

Log mostra qual erro. Checklist:

  • Wazuh URL é válida.
  • Certificado não expirou.
  • Conexão à fila Redis OK.

Pré-deploy checklist​

Antes de deploy com novo mapping/integração:

  • Dispatcher está rodando: docker compose ps | grep dispatcher
  • Wazuh está acessível: curl -k https://wazuh-manager:6514
  • Fila dispatch está vazia: redis-cli LLEN dispatch < 100
  • Evento teste chegou ao Wazuh.

Escalação​

Se problem persiste após steps acima:

  1. Colete evidência:

    docker compose logs dispatcher > dispatcher.log
    docker compose logs collector-worker-priority > workers.log
    docker compose exec redis redis-cli INFO stats > redis.log

  2. Abra GitHub issue com logs acima.

  3. Contate Wazuh support se problema estiver em Wazuh Indexer.

Próximos passos​

  • Evento em quarentena? Vá a Quarentena.
  • Coletando eventos? Vá a Pipeline Health.
  • Wazuh config? Vá a Wazuh Integration.
