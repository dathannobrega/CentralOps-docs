DFIR-IRIS Integration

O CentralOps mapeia cada Organization a um customer no DFIR-IRIS. Esse vínculo (Organization.iris_customer_id) viaja no envelope de todo evento — é ele que diz ao IRIS/Wazuh a qual cliente o evento pertence.

Público: Admin (configurar e vincular), SRE (operar).

:::warning Por que isso é obrigatório Com ENVELOPE_USE_IRIS_CUSTOMER_ID=true (default), uma Organization sem iris_customer_id manda todos os eventos para a Quarentena com o motivo missing_customer_id. Vincular o customer é pré-requisito para a Org entregar eventos. :::

Como funciona​

Diferente de Sophos/Wazuh, o IRIS não é uma integração por-cliente criada na tela de Integrações. É uma configuração global do deployment (uma URL + uma API key), e o vínculo acontece por Organization de dois jeitos:

CaminhoPara quêQuem dispara
AutomáticoOrgs do Sophos Partner (auto-managed)O sync de tenants, ao aprovar
ManualQualquer org (Sophos avulso, Wazuh, criada à mão)Admin, via API

Em ambos a operação é idempotente: o IRIS é consultado por nome antes de criar, então rodar de novo nunca duplica o customer.

Passo 1: Configurar o IRIS (uma vez por deployment)​

O CentralOps autentica no IRIS com uma API key (bearer token). Gere essa key a partir de uma conta de serviço dedicada, com o mínimo de privilégio — nunca a key pessoal de um administrador.

Permissões mínimas da conta de serviço​

O CentralOps faz apenas duas chamadas à API do IRIS, ambas no módulo de customers (ver backend/app/services/iris_client.py):

Operação do CentralOpsEndpoint IRISPermissão exigida
Buscar customer por nome (idempotência)GET /manage/customers/listcustomers_read
Criar customerPOST /manage/customers/addcustomers_write

A conta precisa somente de customers_read + customers_write. Não conceda server_administrator nem permissões de cases/alerts: o CentralOps não cria cases nem alerts no IRIS — ele apenas registra o customer, e o vínculo (iris_customer_id) viaja no envelope para o pipeline materializar depois.

:::info Permissões vêm de Groups No IRIS a permissão não é atribuída direto ao usuário — é herdada da participação em Groups. Crie (ou reutilize) um group com exatamente essas duas permissões e associe a conta de serviço a ele. :::

Criar a conta de serviço e gerar a API key​

  1. Advanced → Access Control → Groups → Add group (ex.: centralops-iris), marcando as permissões customers_read e customers_write.
  2. Advanced → Access Control → Users → Add user. Preencha nome/login/e-mail e marque "Is service account" — contas de serviço não logam na UI e não têm senha; autenticam exclusivamente por API key.
  3. Na aba Groups do usuário, associe-o ao group criado no passo 1.
  4. Abra a conta de serviço em Access Control → Users: a API key aparece no perfil (copiável; "renovar" invalida a anterior). A key herda as permissões do group — esse é o valor de DFIR_IRIS_API_KEY.

Configurar o .env​

DFIR_IRIS_URL=https://iris.exemplo.com
DFIR_IRIS_API_KEY=<API key da conta de serviço>
DFIR_IRIS_TLS_SKIP_VERIFY=false   # true apenas em dev com cert self-signed

Reinicie backend e workers. Sem essas variáveis os dois caminhos ficam desabilitados (o endpoint manual responde 503).

:::info Não há tela para isso no CentralOps A URL/API key do IRIS são lidas do ambiente, não do banco. Trocar exige editar o .env e reiniciar — não há campo na UI do CentralOps. :::

Passo 2: Vincular as Organizations​

Caso A — Sophos Partner (automático)​

Se o IRIS já estava configurado quando o tenant foi descoberto, nada a fazer: o customer é criado ao aprovar o tenant no painel Partner.

Configurou o IRIS depois? Rode um re-sync — ele preenche retroativamente os tenants aprovados que ficaram sem vínculo:

  • UI: painel da integração Partner → Atualizar agora.

  • API:

    curl -X POST "$API/api/integrations/$PARTNER_ID/sync-tenants" \
      -H "Authorization: Bearer $PAT"

O sync diário (04:00 UTC por padrão) faz o mesmo automaticamente.

Caso B — Org manual / não-Partner (forçar)​

Para orgs que não vêm do Partner (o re-sync não toca nelas), force pelo endpoint dedicado:

# Cria/linka o customer IRIS para UMA organization
curl -X POST "$API/api/organizations/$ORG_ID/sync-iris-customer" \
  -H "Authorization: Bearer $PAT"

# Re-vincular (trocou de IRIS ou corrigir vínculo anterior)
curl -X POST "$API/api/organizations/$ORG_ID/sync-iris-customer?force=true" \
  -H "Authorization: Bearer $PAT"

O $PAT é um token (Authorization: Bearer copsk_…) gerado em Settings → API Tokens. O endpoint exige o scope org.manage; o re-sync exige integration.write (ver RBAC).

Respostas do endpoint manual:

CódigoSignificado
200Vinculado — iris_customer_id gravado
409Org já vinculada (use ?force=true para re-vincular)
404org_id não existe
503IRIS não configurado no deployment
502IRIS rejeitou, deu timeout ou não retornou um id válido

Backfill em lote (carteira legada)​

Habilitou o IRIS com a base já cheia? Liste as orgs sem vínculo e itere:

for ORG_ID in 12 34 56; do
  curl -sS -X POST "$API/api/organizations/$ORG_ID/sync-iris-customer" \
    -H "Authorization: Bearer $PAT" -o /dev/null -w "org=$ORG_ID -> %{http_code}\n"
done

:::caution Aprovou tenants pela CLI? O comando bulk-approve aprova os tenants mas não cria o customer no IRIS. Depois de usá-lo, rode o re-sync (Caso A) ou o endpoint manual (Caso B) para vincular. :::

Passo 3: Conferir​

  • Tela de Organizações: orgs vinculadas exibem o badge "Iris #id". A ausência do badge = iris_customer_id ainda NULL.
  • Quarentena: filtre por missing_customer_id — deve cair a zero para as orgs recém-vinculadas.

:::note Eventos já em quarentena Vincular o customer corrige os próximos eventos. Os que já estão em quarentena precisam de Reprocess na tela de Quarentena. :::

Troubleshooting​

"503 — IRIS não configurado"​

DFIR_IRIS_URL/DFIR_IRIS_API_KEY ausentes. Configure no .env e reinicie.

"502 — falha ao sincronizar"​

Causas comuns:
  1. Backend/worker não alcança o DFIR_IRIS_URL (rede, firewall, egress allowlist)
  2. API key inválida, ou a conta de serviço sem customers_read/customers_write
  3. Cert self-signed em dev → DFIR_IRIS_TLS_SKIP_VERIFY=true (nunca em prod)

"Re-sync não vinculou minha org"​

A org é manual (não-Partner)? É esperado — use o Caso B. Caso contrário, confirme que o tenant está aprovado (pending/excluded não materializam).

"Org continua mandando tudo pra quarentena"​

Confirme o vínculo (badge "Iris #id" / SQL iris_customer_id IS NOT NULL) e reprocesse os eventos já em quarentena.
