Guia de Migração: DSL v1 → v2

Histórico: Este documento descrevia a migração de DSL v1 para v2. A partir de 2026-04-28, a versão v1 foi removida do backend. O sistema só aceita DSL v2 ({preprocess, rules}). Mapeamentos v1 são migrados automaticamente no startup. Este documento é mantido como referência histórica.

Como fazer upgrade de um mapeamento v1 existente para v2 para usar novos recursos.

Preciso Migrar?​

Não, se: Seu mapeamento v1 funciona e você não precisa:

  • Pré-processamento (analisar strings JSON aninhadas)
  • Fallback sources (múltiplos campos opcionais)
  • Regras condicionais (pular regra se campo ausente)
  • Array builders (múltiplos observáveis)

Sim, se: Você precisa de qualquer um dos acima. Mapeamentos v1 continuarão funcionando para sempre; sem migração forçada.

O que há de Novo em v2​

Recursov1v2Caso de uso
Bloco Preprocess✗✓Analisar processedData antes de regras
fallback_source✗✓Tentar múltiplos campos (createdAt ou raisedAt)
Predicados when✗✓Pular regra se campo ausente
array_builder✗✓Construir arrays de observáveis
expected_always_default✗✓Marcar regras de placeholder intencional

Tudo mais (source, const, default, value_map, type_cast, pre_cast, required) funciona identicamente em v1 e v2.

Etapas de Migração​

Passo 1: Identifique Por que Você Precisa de v2​

Problema: Meu mapeamento tem esta regra:

{
  "target": "normalized.severity_id",
  "source": "severity",
  "value_map": { "critical": 5, "high": 4, "medium": 3, "low": 2, "info": 1 }
}

Mas às vezes severidade está em um campo diferente: urgency.

Solução: Use fallback_source (recurso v2).

Passo 2: Altere a Forma de Mapeamento​

Forma v1 (lista):

{
  "rules": [ ... ]
}

Forma v2 (dicionário):

{
  "preprocess": [ ... ],  // Opcional; omita se não necessário
  "rules": [ ... ]
}

Passo 3: Adicione Pré-processamento (Se Necessário)​

Se seu vendor envolve dados em strings JSON (como processedData do Sophos), adicione pré-processamento:

{
  "preprocess": [
    {
      "op": "json_parse",
      "source": "processedData",
      "target": "_processed",
      "tolerant": true
    }
  ],
  "rules": [
    // Agora regras podem usar "_processed.parsedAlert.fields.clientIp" etc.
  ]
}

Passo 4: Atualize Regras​

Antes (v1):

{
  "target": "normalized.severity_id",
  "source": "severity",
  "value_map": { "critical": 5, "high": 4 }
}

Depois (v2, com fallback):

{
  "target": "normalized.severity_id",
  "source": "severity",
  "fallback_source": ["urgency"],  // Tente urgency se severity for null
  "value_map": { "critical": 5, "high": 4 }
}

Depois (v2, com condicional):

{
  "target": "normalized.severity_id",
  "source": "severity",
  "value_map": { "critical": 5, "high": 4 },
  "when": { "exists": "severity" }  // Apenas executar se severity existe
}

Passo 5: Adicione Array Builders (Se Necessário)​

Abordagem v1 antiga (plana):

{
  "target": "normalized.email_recipients",
  "source": "recipients",  // Assuma email único
  "default": []
}

Nova abordagem v2 (observáveis):

{
  "target": "normalized.observables",
  "kind": "array_builder",
  "items": [
    {
      "name": "email_to",
      "type": "Email Address",
      "type_id": 5,
      "source": "recipients",
      "explode": true  // Múltiplos destinatários → um observável cada
    }
  ]
}

Passo 6: Salve e Teste​

  1. Abra seu mapeamento na UI do CentralOps.
  2. Cole o novo JSON de forma v2.
  3. Clique Save → cria novo MappingVersion com dsl_version: 2.
  4. Execute Dry-Run para verificar que todos os eventos ainda mapeiam corretamente.
  5. Se houver erros, verifique Guia de Resolução de Problemas.

Real Example: Sophos Detection​

Original v1 Mapping (Excerpt)​

{
  "rules": [
    { "target": "normalized.class_uid", "const": 2004, "required": true },
    { "target": "normalized.class_name", "const": "Detection Finding" },
    { "target": "normalized.time", "source": "time", "type_cast": "iso_to_epoch", "required": true },
    { "target": "normalized.finding_info.uid", "source": "id", "required": true },
    {
      "target": "normalized.severity_id",
      "source": "severity",
      "type_cast": "to_int",
      "default": 0,
      "required": true
    }
  ]
}

Problem: Nested email data in processedData string can't be extracted.

Migrated v2 Mapping​

{
  "preprocess": [
    {
      "op": "json_parse",
      "source": "processedData",
      "target": "_processed",
      "tolerant": true
    }
  ],
  "rules": [
    { "target": "normalized.class_uid", "const": 2004, "required": true },
    { "target": "normalized.class_name", "const": "Detection Finding" },
    { "target": "normalized.time", "source": "time", "type_cast": "iso_to_epoch", "required": true },
    { "target": "normalized.finding_info.uid", "source": "id", "required": true },
    {
      "target": "normalized.severity_id",
      "source": "severity",
      "type_cast": "to_int",
      "default": 0,
      "required": true
    },
    // NEW: Extract email data from preprocessed JSON
    {
      "target": "normalized.observables",
      "kind": "array_builder",
      "items": [
        {
          "name": "src_ip",
          "type": "IP Address",
          "type_id": 2,
          "source": "_processed.parsedAlert.fields.clientIp",
          "skip_null": true
        },
        {
          "name": "email_to",
          "type": "Email Address",
          "type_id": 5,
          "source": "_processed.parsedAlert.fields.envelopeRecipients",
          "explode": true,
          "skip_null": true
        }
      ]
    }
  ]
}

Mudanças:

  1. Empacotado num dict com chaves preprocess e rules.
  2. Adicionado preprocess para parsear processedData em _processed.
  3. Adicionado array_builder para extrair múltiplos observables dos dados aninhados de _processed.

Histórico de versões​

Quando você salva um mapping v2, o CentralOps mantém o histórico:

  • Versão 1 (v1): mapping original em formato lista.
  • Versão 2 (v2): mapping novo em formato dict com preprocess.

Ativando uma versão:

  • Apenas uma versão fica ativa por vez para um par collector+event_type.
  • Ative a Versão 2 → todos os eventos novos passam a usar as regras v2.
  • Se v2 tem bug, faça rollback para a Versão 1 (veja Troubleshooting).

Consultar a versão ativa:

GET /api/mappings/normalize?collector=sophos_detection&active=true

Rollback para v1​

Se a migração v2 causar problemas:

  1. Vá em Mappings → [Seu Mapping] → Versões.
  2. Clique Ativar na versão v1.
  3. Eventos novos passam a usar as regras v1 imediatamente.
  4. Corrija o mapping v2 e salve como Versão 3.

Padrões comuns de migração​

Padrão 1: Nome do campo do vendor varia​

Antes (v1, sem fallback):

{ "target": "normalized.time", "source": "createdAt", "required": true }

Vendor às vezes manda raisedAt no lugar. v1 não consegue expressar "tente createdAt, com fallback para raisedAt".

Depois (v2):

{
  "target": "normalized.time",
  "source": "createdAt",
  "fallback_source": ["raisedAt", "processedAt"],
  "type_cast": "iso_to_epoch",
  "required": true
}

Padrão 2: String JSON aninhada​

Antes (v1, não consegue parsear):

{
  "target": "normalized.src_endpoint.ip",
  "source": "jsonData.ip",  // Não funciona se jsonData é uma string
  "default": null
}

Depois (v2, com preprocess):

{
  "preprocess": [
    { "op": "json_parse", "source": "jsonData", "target": "_data", "tolerant": true }
  ],
  "rules": [
    { "target": "normalized.src_endpoint.ip", "source": "_data.ip", "default": null }
  ]
}

Padrão 3: Múltiplos destinatários​

Antes (v1, lista flat):

{ "target": "normalized.email_recipients", "source": "recipients", "default": [] }

Saída: ["user1@domain", "user2@domain"] — não está no formato OCSF observables.

Depois (v2, observables):

{
  "target": "normalized.observables",
  "kind": "array_builder",
  "items": [
    {
      "name": "email_to",
      "type": "Email Address",
      "type_id": 5,
      "source": "recipients",
      "explode": true
    }
  ]
}

Saída: array de observables conforme OCSF, um observable por destinatário.

Padrão 4: Extração condicional​

Antes (v1, sempre escreve):

{ "target": "normalized.actor.user.email", "source": "mailFrom", "default": null }

Mesmo que mailFrom seja null, o campo é escrito com valor null.

Depois (v2, pula se ausente):

{
  "target": "normalized.actor.user.email",
  "source": "mailFrom",
  "when": { "exists": "mailFrom" }
}

O campo é omitido da saída se mailFrom for null.

Troubleshooting da migração​

"DSL v2 shape expected dict with 'preprocess' and 'rules', got list"​

Causa: você colou o formato lista (v1) na UI esperando que o v2 parseasse.

Correção: envolva num dict:

// Errado (formato v1)
[ { "target": "...", "source": "..." } ]

// Correto (formato v2)
{
  "preprocess": [],
  "rules": [ { "target": "...", "source": "..." } ]
}

"target begins with '_' but not in preprocess namespace"​

Causa: uma regra tenta usar _field como target, namespace reservado para saídas de preprocess.

Correção: ou use preprocess para popular o _field, ou renomeie o target:

// Errado (namespace reservado)
{ "target": "_my_field", "source": "..." }

// Correto (se quer pré-processar)
{
  "preprocess": [ { "op": "json_parse", "source": "data", "target": "_my_field" } ],
  "rules": [ { "target": "normalized.field", "source": "_my_field" } ]
}

// Correto (se não precisa de preprocess)
{ "target": "normalized.my_field", "source": "..." }

Regras com when não executam em v1​

Causa: v1 não suporta predicados when.

Correção: confirme que está usando o formato v2 (dict com preprocess e rules).

Testando a migração​

  1. Salve o novo mapping v2 (cria nova versão).
  2. Rode dry-run numa amostra representativa (10-20 eventos).
  3. Verifique erros: alguma quarentena? Provavelmente problema de regra; veja o Guia de Troubleshooting.
  4. Verifique drift: o relatório do dry-run mostra campos não mapeados; confirme se são intencionais.
  5. Ative v2 → eventos novos passam a usar v2.
  6. Monitore por 1 hora: verifique se há pico de quarentena. Se OK, concluído.
  7. Se houver problema, rollback para v1: clique Ativar na Versão 1.

FAQ​

P: Posso usar mappings v1 e v2 simultaneamente?
R: Apenas uma versão fica ativa por par collector/event_type. Troque clicando em Ativar. Eventos já processados com a versão antiga permanecem inalterados.

P: A migração quebra dados existentes?
R: Não. Eventos anteriores já foram normalizados. Apenas os eventos novos usam o mapping novo.

P: Como sei qual versão está ativa?
R: Vá em Mappings → [Seu Mapping] → Versões. A versão ativa tem um marcador verde.

P: Posso fazer downgrade de v2 para v1?
R: Sim, clique Ativar na versão v1. Você perde as features v2 mas as regras continuam funcionando.

P: Tem v3?
R: Ainda não. v2 é a versão atual. Novas features (se houver) serão documentadas em guias futuros.
