Guia de Resolução de Problemas

Guia de diagnóstico orientado por sintoma para problemas de mapeamento.

Eventos em Quarentena com error_kind: "map"​

Evento falhou ao mapear e é marcado como quarentena. O campo error_message contém o motivo.

Sintoma: required field 'normalized.time' resolved to None​

Diagnóstico:

  • Uma regra com required: true resolveu para null.
  • Geralmente um campo source ausente ou vazio, ou um type_cast que falhou.

Verificar:

  1. O evento do vendor tem o campo? (Olhe uma amostra de evento.)
  2. A expressão JMESPath está correta? Teste: jmespath.search(expression, event).
  3. O type_cast está rejeitando entrada válida? (Ex: to_int("abc") falha.)

Opções de correção:

  1. Campo às vezes ausente → adicione default:

    { "target": "normalized.time", "source": "timestamp", "default": 0, "required": true }

  2. Nome de campo varia → use fallback_source:

    {
      "target": "normalized.time",
      "source": "createdAt",
      "fallback_source": ["raisedAt", "processedAt"],
      "required": true
    }

  3. Incompatibilidade de tipo → adicione pre_cast ou value_map:

    {
      "target": "normalized.severity_id",
      "source": "severity",
      "pre_cast": "to_str",          // Converter para string primeiro
      "value_map": { "high": 4, "low": 2 },
      "required": true
    }

  4. Campo é verdadeiramente opcional → remova required:

    { "target": "normalized.device.name", "source": "device.name", "default": null }

Sintoma: iso_to_epoch: timestamp invalid "2026-04-27T12:30:00+05:30"​

Diagnóstico:

  • type_cast: "iso_to_epoch" recebeu um timestamp mas falhou ao analisá-lo.
  • Motivo: Vendor usa offset ou formato de timezone incomum.

Verificar:

  1. Como é o timestamp do vendor? (Ex: "2026-04-27T12:30:00+05:30" com offset não-padrão?)
  2. É realmente compatível com ISO-8601?

Opções de correção:

  1. Offset do vendor não-padrão → use trim + preprocess:

    {
      "target": "normalized.time",
      "source": "timestamp",
      "pre_cast": "trim",
      "type_cast": "iso_to_epoch"
    }

  2. Vendor envia milissegundos em vez de segundos → pré-processa para corrigir: Se timestamp é 1746014700000 (milissegundos), pré-processamento ainda não está disponível para transformações numéricas. Solução alternativa: verifique se vendor envia epoch em milissegundos ou segundos. Se milissegundos, documente e sinalize para um aprimoramento futuro.

  3. Formato de timestamp não-ISO → solicite ao vendor normalizar ou adicione um cast customizado.

Symptom: to_int: string non-numeric "N/A"​

Diagnóstico:

  • type_cast: "to_int" recebeu uma string que não é número.
  • Vendor às vezes manda placeholder como "N/A" ou "unknown".

Verifique:

  1. Com que frequência o vendor manda inteiros inválidos? (Veja a distribuição no drift explorer.)
  2. A regra deve pular eventos com inteiros inválidos ou usar um default?

Opções de correção:

  1. Vendor manda "N/A" raramente → use default:

    {
      "target": "normalized.severity_id",
      "source": "severity",
      "type_cast": "to_int",
      "default": 0
    }

  2. Mapeie strings para ints ANTES do type_cast:

    {
      "target": "normalized.severity_id",
      "source": "severity",
      "value_map": { "critical": 5, "high": 4, "N/A": 0 },
      "type_cast": "to_int",
      "default": 0
    }

  3. Use regra condicional para pular se inválido:

    {
      "target": "normalized.severity_id",
      "source": "severity",
      "type_cast": "to_int",
      "when": { "in": { "source": "severity", "values": ["1", "2", "3", "4", "5"] } }
    }

Sintoma: score_to_percent: int 150 out of range [0, 100]​

Diagnóstico:

  • Vendor manda severity: 150 (escala 0-150).
  • Você usou type_cast: "score_to_percent" que espera 0-100 ou 0-1.

Verifique:

  1. Qual a escala de severity do vendor? (0-10, 0-100, 0-1000, customizada?)
  2. Deveria mapear para OCSF 1-6?

Opções de correção:

  1. Vendor é 0-150 → normalize para 0-100 primeiro:

    {
      "target": "normalized.confidence",
      "source": "severity",
      "value_map": {
        // Mapeia 0-150 para faixas 0-100
        "0": 0, "50": 33, "100": 66, "150": 100
      },
      "type_cast": "score_to_percent"  // Agora converte int 0-100 (idempotente)
    }

  2. Pule score_to_percent, use to_int direto:

    {
      "target": "normalized.confidence",
      "source": "severity",
      "type_cast": "to_int"  // Passagem direta
    }

Sintoma: required field 'normalized.observables' resolved to None (array_builder)​

Diagnóstico:

  • Regra array_builder (kind: "array_builder") teve todos os items resolvidos para null e skip_null: true produziu array vazio.
  • required: true em array_builder não é permitido (required em nível de regra é proibido para array builders).

Verifique:

  1. Você marcou a regra como required: true por engano? (Não é permitido.)
  2. Todos os observable items estão ausentes no evento do vendor?

Opções de correção:

  1. Remova required do array_builder (não suportado):

    {
      "target": "normalized.observables",
      "kind": "array_builder",
      "items": [ ... ],
      // NÃO use "required: true" aqui
    }

  2. Se precisa de pelo menos um observable, adicione regra condicional para setar default:

    {
      "target": "normalized.observables",
      "kind": "array_builder",
      "items": [
        {
          "name": "device_id",
          "type": "Resource UID",
          "type_id": 33,
          "source": "device.id"
        }
      ],
      "skip_null": true,
      "when": { "exists": "device.id" }  // Só escreve se device ID existir
    }

Dry-Run Mostra Aviso "100% Default"​

Dry-run escaneou seu mapeamento contra eventos de amostra e encontrou uma regra que sempre usa o valor padrão (source sempre null).

É isso um bug?​

Provavelmente não, se:

  • Vendor não suporta o campo (gap documentado).
  • Regra é intencionalmente um placeholder para uso futuro.
  • Fallback sources são todos opcionais.

Provavelmente é, se:

  • Nome de campo está malescrito.
  • Expressão JMESPath tem erro de sintaxe (mostraria em tempo de compilação, não dry-run).
  • Você esqueceu a etapa de pré-processamento (ex: rule sources _processed.* mas nenhum pré-processamento definido).

Como corrigir​

Opção 1: Confirme que é intencional

{
  "target": "normalized.src_endpoint.ip",
  "source": "sourceIp",
  "default": null,
  "expected_always_default": true  // Suprimir aviso
}

Opção 2: Remova a regra Se o campo não é importante, delete-o completamente do mapeamento.

Opção 3: Depure a source Verifique se o evento do vendor realmente tem o campo:

import jmespath
event = {...}
result = jmespath.search("sourceIp", event)
print(result)  # Se None, campo não existe

Se JMESPath retorna None, a expressão está correta mas vendor não a fornece. Adicione expected_always_default: true.

Drift Explorer mostra muitos campos não mapeados​

O relatório do Drift Explorer lista campos do vendor que seu mapping não usa.

Padrões comuns​

Campos descartáveis (ignorar):

  • _internal_id, _metadata, _debug, __v (marcadores de versão)
  • UUIDs opacos sem significado semântico
  • Campos que duplicam informação (ex: tanto raw_json quanto os campos já parseados)

Lacunas reais (considere adicionar):

  • Endereços IP, hostnames, usernames
  • Caminhos de arquivo, hashes, domínios
  • Severity, tipo de detecção, categoria
  • Timestamps, timezones
  • Detalhes de processo/arquivo/rede

Fluxo​

  1. Revise os campos não mapeados no Drift Explorer.
  2. Pergunte: "Isso é acionável numa investigação de segurança?"
  3. Decida:
    • Sim → Adicione uma regra. Pode ser via observables ou regras escalares simples.
    • Não → Marque como descartável. Adicione um comentário no mapping explicando o porquê.

Exemplo de comentário:

// _internal_id: ID interno do Sophos, não acionável para o SOC. Não mapeado intencionalmente.

Mudança de mapping quebrou produção (Rollback)​

Você publicou uma versão nova do mapping e os eventos começaram a quarentenar.

Passos para rollback​

  1. Identifique a versão ruim: Na UI do CentralOps, vá em Mappings → [Seu Mapping] → Versões.
  2. Anote o número da versão que funcionava (a anterior à última alteração).
  3. Ative a versão anterior: Clique em "Ativar" na versão antiga.
  4. Efeito imediato: Eventos novos passam a usar o mapping antigo.
  5. Eventos em quarentena: Eventos que ficaram presos durante a versão ruim continuam lá. Você precisa replay manual depois de corrigir o mapping.

Prevenir regressão​

  1. Sempre teste localmente com dry-run antes de publicar.
  2. Teste com uma amostra representativa de eventos do vendor (não só um).
  3. Se a mudança é grande, publique em estágios:
    • Publique a regra nova de campo opcional primeiro.
    • Depois, altere o mapping de campo required (risco maior).

DSL v2 espera dict, recebeu list​

A compilação do mapping falha com: "DSL v2 shape expected dict with 'preprocess' and 'rules', got list".

Diagnóstico:

  • Você tentou usar features v2 (preprocess, fallback_source, when) mas envolveu o mapping numa lista.

Errado (formato v1):

[
  { "preprocess": [ ... ] },  // ERRO: preprocess no formato lista
  { "target": "...", "source": "..." }
]

Correto (formato v2):

{
  "preprocess": [ ... ],
  "rules": [
    { "target": "...", "source": "..." }
  ]
}

Etapa de preprocess falha silenciosamente com tolerant: true​

Operador de preprocess (ex: json_parse) falhou em silêncio porque tolerant: true, e agora as regras seguintes não acham os campos esperados.

Exemplo​

Mapping diz:

{
  "preprocess": [
    { "op": "json_parse", "source": "processedData", "target": "_processed", "tolerant": true }
  ],
  "rules": [
    { "target": "normalized.src_endpoint.ip", "source": "_processed.parsedAlert.fields.clientIp" }
  ]
}

Se processedData contém JSON malformado:

  1. json_parse com tolerant: true retorna None.
  2. _processed vira None.
  3. JMESPath _processed.parsedAlert.fields.clientIp em None retorna None.
  4. Regra cai no default.

Correção:

  1. Adicione um predicado when para pular a regra se o preprocess falhou:

    { "target": "normalized.src_endpoint.ip", "source": "_processed.parsedAlert.fields.clientIp", "when": { "exists": "_processed" } }

  2. Ou inspecione o JSON do vendor. Se é frequentemente malformado, use tolerant: false para pegar o problema cedo.

Campo aparece em alguns eventos e em outros não​

Vendor às vezes inclui o campo, às vezes não.

Exemplo: detection.ruleId existe em 70% das detecções Sophos, null nos outros 30%.

Estratégias de correção:

  1. Use fallback_source com várias opções:

    {
      "target": "normalized.finding_info.uid",
      "source": "id",
      "fallback_source": ["eventId"],
      "required": true
    }

  2. Use regra condicional (predicado when):

    {
      "target": "normalized.finding_info.uid",
      "source": "ruleId",
      "when": { "exists": "ruleId" }
    }

  3. Combine com default:

    {
      "target": "normalized.finding_info.uid",
      "source": "ruleId",
      "default": "unknown",
      "required": true
    }

Expressão JMESPath não acha o dado​

Você escreveu source: "device.name" mas o valor vem null mesmo que o evento do vendor tenha o dado.

Causas comuns:

  1. Sensibilidade a case: O campo do vendor é Device.name (D maiúsculo), não device.name.

    "source": "Device.name"  // Correto

  2. Falta índice de array: Vendor manda tags: ["tag1", "tag2"] mas você escreveu source: "tags".

    "source": "tags[0]"  // Pega a primeira tag
    // OU
    "source": "[tags]"   // Embrulha o array em outro array (raro)

  3. Erro de digitação no caminho aninhado: O caminho do vendor é device.specs.cpu, não device.cpu.

    "source": "device.specs.cpu"  // Correto

  4. Vendor usa estrutura diferente em tipos de evento diferentes: Alguns eventos têm device.name, outros têm machine.hostname.

    "source": "device.name",
    "fallback_source": ["machine.hostname"]

Debug: Use um REPL Python para testar:

import jmespath
event = { ... }  # Evento do vendor
expr = "device.name"
result = jmespath.search(expr, event)
print(f"Resultado: {result}")

Value map não casa​

Você definiu value_map: { "high": 4, "low": 2 } mas os eventos passam direto sem mapear.

Causas comuns:

  1. Sensibilidade a case (raro para strings em value_map): Chaves de value_map são case-insensitive, então isso não deveria acontecer. Mas se acontecer, normalize o case:

    {
      "target": "normalized.severity_id",
      "source": "severity",
      "pre_cast": "lowercase",  // Normaliza para minúsculas primeiro
      "value_map": { "high": 4, "low": 2 }
    }

  2. Tipos diferentes: A source vem como int 4, mas as chaves do value_map são strings "high", "low".

    {
      "target": "normalized.severity_id",
      "source": "severity",
      "pre_cast": "to_str",      // Converte int para string
      "value_map": { "4": 4, "2": 2 }  // Mapeia pela representação string
    }

  3. Valor fora do mapa → passa direto (comportamento intencional): Se o vendor manda severity "unknown" e ele não está no mapa, passa direto como "unknown". Se não quer isso, adicione um catch-all:

    "value_map": { "high": 4, "low": 2, "unknown": 0 }  // Default explícito para valores desconhecidos

Como ler mensagens de erro​

ErroCausa provávelOnde olhar
required field 'X' resolved to NoneCampo obrigatório ausente ou type_cast falhouExpressão source, valor de default
iso_to_epoch: timestamp invalid "..."Formato de timestamp não suportadoFormato do vendor; use pre_cast ou trim
to_int: string non-numeric "..."Vendor mandou não-número em campo numéricoAdicione value_map ou default
unknown type_cast "..."Erro de digitação no nome do castConfira o nome; veja a Referência de Operadores
JMESPath invalid ...Erro de sintaxe na expressão sourceVerifique sintaxe JMESPath; teste em Python
'when' predicate requires DSL v2Usando feature v2 (when) em mapping v1Mude o formato de list para dict
target begins with '_' but not in preprocessRegra usa campo _extracted sem preprocessAdicione regra de preprocess OU mude target para campo raw

Onde buscar ajuda​

  1. Confira o Drift Explorer para campos do vendor não mapeados.
  2. Confira a saída do dry-run para regras que sempre caem no default.
  3. Teste expressões JMESPath em Python: jmespath.search(expr, event).
  4. Consulte a Referência de Operadores para detalhes de cast/predicado.
  5. Olhe mappings de vendors parecidos (Sophos alerts e detections) para padrões.
