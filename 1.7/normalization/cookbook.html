Cookbook de Normalização

Doze receitas para cenários comuns de mapeamento. Cada uma mostra entrada, regra DSL, saída e explicação.

Receita 1: Severity Int → OCSF Int​

Problema: Detecção do Sophos envia severity: 0..10 (escala do vendor), OCSF espera severity_id: 1..6.

Contexto do vendor: Eventos de Detecção do Sophos XDR.

Evento de entrada (recorte)​

{
  "id": "det_001",
  "time": "2026-04-27T12:30:00Z",
  "severity": 7,
  "detectionRule": "Malware.Generic"
}

Regra(s) de mapeamento​

{
  "target": "normalized.severity_id",
  "source": "severity",
  "type_cast": "to_int",
  "required": true
}

O que isso produz​

{
  "normalized": {
    "severity_id": 7
  }
}

Por que funciona​

A source extrai o int diretamente do evento do vendor. to_int garante que permaneça um int (idempotente para ints). A escala do vendor (0-10) é preservada como está porque não há value_map para transformá-la. Se você precisa redimensionar para OCSF 1-6, adicione um value_map com entradas explícitas.

Nota: Se seu vendor envia severity como uma string como "7", adicione pre_cast: "to_int" antes desta regra.

Variações / pegadinhas​

  • Vendor às vezes omite severity? Adicione default: 0.
  • Escala do vendor não corresponde a OCSF? Use value_map com entradas como "7": 4 (Sophos 7 → OCSF 4).

Receita 2: Severity String → OCSF Int​

Problema: Alerta do Sophos envia severity: "critical" (string do vendor), OCSF espera severity_id: 5 (int).

Contexto do vendor: Eventos de Alerta do Sophos (firewall, EDR).

Evento de entrada (recorte)​

{
  "id": "alert_001",
  "severity": "high",
  "description": "Intrusion attempt blocked"
}

Regra(s) de mapeamento​

{
  "target": "normalized.severity_id",
  "source": "severity",
  "value_map": {
    "critical": 5,
    "high": 4,
    "medium": 3,
    "low": 2,
    "info": 1,
    "informational": 1
  },
  "default": 0,
  "required": true
}

O que isso produz​

{
  "normalized": {
    "severity_id": 4
  }
}

Por que funciona​

source extrai "high" do evento. Não é necessário pre_cast porque chaves string em value_map correspondem a strings diretamente. A busca encontra "high" → 4. Se severity estivesse completamente ausente, default: 0 seria usado.

Nota específica do vendor: Chaves de severity de alerta do Sophos são case-sensitive em minúsculas; se você vê caso misto, adicione pre_cast: "lowercase" antes de value_map.

Variações / pegadinhas​

  • Case-sensitivity: chaves de value_map são case-insensitive para strings.
  • Severity desconhecida no evento? Default é usado; sem erro a menos que required: true e default seja null.
  • Quer adicionar severidades customizadas? Apenas adicione mais entradas ao mapa.

Receita 3: Título com Múltiplos Fallbacks​

Problema: Diferentes tipos de detecção têm título em campos diferentes. Detecções de email do Sophos usam ruleDescription, detecções de endpoint usam attackType.

Contexto do vendor: Eventos de Detecção do Sophos XDR (multi-fonte).

Evento de entrada (recorte — variante de email)​

{
  "id": "det_002",
  "detectionRule": "Email.Phishing.Advanced",
  "attackType": "Phishing",
  "ruleDescription": "Advanced phishing with graphics spoofing"
}

Regra(s) de mapeamento​

{
  "target": "normalized.finding_info.title",
  "source": "detectionRule",
  "fallback_source": ["attackType"],
  "default": "Unknown Detection"
}

O que isso produz​

{
  "normalized": {
    "finding_info": {
      "title": "Email.Phishing.Advanced"
    }
  }
}

Por que funciona​

O mecanismo tenta detectionRule primeiro. Se for null, tenta attackType. Se ambos forem null, usa "Unknown Detection". Este é um recurso v2 — mapeamentos v1 não podem usar fallbacks.

Na prática: Detecção do Sophos sempre tem detectionRule, então fallback é seguro. Se seu vendor é inconsistente, fallback garante que você sempre obtenha algo.

Variações / pegadinhas​

  • Todos os fallbacks devem usar a mesma raiz de source (ambas raw ou ambas _extracted). Sem mistura.
  • Cadeias de fallback são tentadas da esquerda para a direita; primeiro não-null vence.
  • Para lógica mais complexa, combine com predicados when (Receita 6).

Receita 4: Abrir JSON Aninhado Compactado​

Problema: Detecção do Sophos envolve dados de alerta de email como uma string-de-JSON em processedData. Você precisa extrair campos aninhados como mailFrom, clientIp.

Contexto do vendor: Eventos de Detecção do Sophos XDR (email).

Evento de entrada (recorte)​

{
  "id": "det_003",
  "time": "2026-04-27T12:30:00Z",
  "severity": 7,
  "processedData": "{\"parsedAlert\":{\"fields\":{\"clientIp\":\"192.0.2.1\",\"mailFrom\":\"attacker@evil.com\",\"envelopeRecipients\":[\"victim@company.com\"]}}}"
}

Regra(s) de mapeamento​

{
  "preprocess": [
    {
      "op": "json_parse",
      "source": "processedData",
      "target": "_processed",
      "tolerant": true
    }
  ],
  "rules": [
    {
      "target": "normalized.src_endpoint.ip",
      "source": "_processed.parsedAlert.fields.clientIp",
      "default": null
    },
    {
      "target": "normalized.actor.user.email_addr",
      "source": "_processed.parsedAlert.fields.mailFrom",
      "default": null
    }
  ]
}

O que isso produz​

{
  "normalized": {
    "src_endpoint": {
      "ip": "192.0.2.1"
    },
    "actor": {
      "user": {
        "email_addr": "attacker@evil.com"
      }
    }
  }
}

Por que funciona​

Etapa de pré-processamento: json_parse lê a string processedData, a analisa como JSON e armazena o resultado em _processed (namespace reservado para campos extraídos).

Etapa de regras: As regras subsequentes agora podem usar JMESPath em _processed como se fosse um objeto aninhado normal. O caminho separado por ponto _processed.parsedAlert.fields.clientIp é exatamente o que o JMESPath precisa.

tolerant: true: Se a string JSON está malformada, análise falha silenciosamente e retorna null (sem quarentena). Se false (padrão), JSON malformado lança um erro e quarentena o evento.

Variações / pegadinhas​

  • tolerant: true é recomendado para dados de vendor com problemas ocasionais de codificação.
  • Se sua string JSON é muito grande (> 1 MB), json_parse a rejeita como proteção DDoS. Aumente a variável de ambiente PREPROCESS_JSON_MAX_BYTES se necessário.
  • Esqueceu o bloco de pré-processamento? Regras falharão com "source começa com _ mas nenhum pré-processamento encontrado."

Receita 5: Construir Observáveis de Múltiplas Fontes​

Problema: Você precisa produzir um array observables com múltiplos itens: IP de origem, endereços de email (múltiplos destinatários), hashes de arquivo e nomes de anexos. Cada um vem de um caminho diferente no evento do vendor.

Contexto do vendor: Detecção do Sophos XDR (email).

Evento de entrada (recorte — após pré-processamento)​

{
  "_processed": {
    "parsedAlert": {
      "fields": {
        "clientIp": "192.0.2.1",
        "mailFrom": "attacker@evil.com",
        "envelopeRecipients": ["victim1@company.com", "victim2@company.com"],
        "attachments": [
          { "name": "invoice.exe", "checksum": "abc123def456" },
          { "name": "document.pdf", "checksum": "ghi789jkl012" }
        ]
      }
    }
  }
}

Mapping rule(s)​

{
  "target": "normalized.observables",
  "kind": "array_builder",
  "items": [
    {
      "name": "src_ip",
      "type": "IP Address",
      "type_id": 2,
      "source": "_processed.parsedAlert.fields.clientIp",
      "skip_null": true
    },
    {
      "name": "email_from",
      "type": "Email Address",
      "type_id": 5,
      "source": "_processed.parsedAlert.fields.mailFrom",
      "skip_null": true
    },
    {
      "name": "email_to",
      "type": "Email Address",
      "type_id": 5,
      "source": "_processed.parsedAlert.fields.envelopeRecipients",
      "explode": true,
      "skip_null": true
    },
    {
      "name": "file_hash",
      "type": "Hash",
      "type_id": 8,
      "source": "_processed.parsedAlert.fields.attachments[*].checksum",
      "explode": true,
      "skip_null": true
    },
    {
      "name": "file_name",
      "type": "File Name",
      "type_id": 7,
      "source": "_processed.parsedAlert.fields.attachments[*].name",
      "explode": true,
      "skip_null": true
    }
  ],
  "skip_null": true,
  "dedup_by": ["value"]
}

O que isso produz​

{
  "normalized": {
    "observables": [
      { "name": "src_ip", "type": "IP Address", "type_id": 2, "value": "192.0.2.1" },
      { "name": "email_from", "type": "Email Address", "type_id": 5, "value": "attacker@evil.com" },
      { "name": "email_to", "type": "Email Address", "type_id": 5, "value": "victim1@company.com" },
      { "name": "email_to", "type": "Email Address", "type_id": 5, "value": "victim2@company.com" },
      { "name": "file_hash", "type": "Hash", "type_id": 8, "value": "abc123def456" },
      { "name": "file_hash", "type": "Hash", "type_id": 8, "value": "ghi789jkl012" },
      { "name": "file_name", "type": "File Name", "type_id": 7, "value": "invoice.exe" },
      { "name": "file_name", "type": "File Name", "type_id": 7, "value": "document.pdf" }
    ]
  }
}

Por que funciona​

array_builder: Tipo de regra especial que produz um array de objetos em vez de um valor escalar único.

explode: Quando verdadeiro, se a source resolver para uma lista, um observável é criado por elemento. Exemplo: envelopeRecipients: ["user1", "user2"] explode em dois observáveis (um por destinatário).

skip_null: Tanto no nível de regra (padrão true) quanto no nível de item. Se uma source de item é null, esse item é omitido. Regras skip_null: true significa que o array inteiro é omitido se está vazio.

dedup_by: Depois que todos os itens são coletados, remover duplicatas baseado no campo value (primeira ocorrência vence). Isso previne IPs ou emails duplicados de serem listados duas vezes.

JMESPath [*]: A sintaxe attachments[*].checksum significa "para cada anexo na lista, extrair o campo checksum." Combinado com explode: true, isso cria um observável por hash.

Variações / pegadinhas​

  • Lista ausente: Se envelopeRecipients é null, nenhum observável é produzido para esse item. Se é uma string única em vez de uma lista, explode: true a envolve em uma lista de um elemento.
  • Anexos vazios: Se não há anexos, attachments[*].checksum retorna uma lista vazia, não produzindo observáveis.
  • Ressalva de dedup: dedup_by: ["value"] deduplica no campo value do observável. Você também pode dedup em outros campos como ["type_id", "value"] para controle mais granular.

Receita 6: Regra Condicional com Predicado when​

Problema: Você quer definir actor.user.email_addr apenas se o vendor realmente forneceu um endereço de email. Caso contrário, omita o campo.

Contexto do vendor: Alerta do Sophos (algumas variantes têm email do usuário, outras não).

Evento de entrada (recorte)​

{
  "id": "alert_001",
  "description": "Intrusion blocked",
  "person": {
    "name": "John Doe",
    "email": "john@company.com"
  }
}

Regra(s) de mapeamento​

{
  "target": "normalized.actor.user.email_addr",
  "source": "person.email",
  "when": { "exists": "person.email" }
}

O que isso produz (se email existe)​

{
  "normalized": {
    "actor": {
      "user": {
        "email_addr": "john@company.com"
      }
    }
  }
}

O que isso produz (se email ausente)​

{
  "normalized": {}
}

Por que funciona​

when: { exists: ... }: A regra é executada apenas se a expressão JMESPath em exists resolver para um valor não-null. Se person.email é null ou ausente, a regra inteira é pulada — o campo de target não é escrito de forma alguma.

Diferença semântica de default: Usar default: null ainda escreveria actor.user.email_addr: null na saída. Usar when: { exists: ... } omite o campo completamente.

Variações / pegadinhas​

  • exists vs equals: exists verifica por não-null. equals verifica um valor específico. in verifica pertencimento em uma lista.
  • Negação: Use when: { not: { exists: "person.email" } } para inverter a lógica (executar se email estiver ausente).
  • Múltiplas condições: Não suportado ainda; use regras separadas se você precisa de lógica AND/OR.

Receita 7: Predicado de Negação​

Problema: Defina um campo apenas se outro campo NÃO é um valor sentinela como "unknown".

Contexto do vendor: Vendor envia campo de direção mas às vezes o preenche com placeholder.

Evento de entrada (recorte)​

{
  "id": "flow_001",
  "direction": "outbound"
}

Regra(s) de mapeamento​

{
  "target": "normalized.traffic.direction",
  "source": "direction",
  "when": { "not": { "equals": { "source": "direction", "value": "unknown" } } }
}

O que isso produz (se direção não é "unknown")​

{
  "normalized": {
    "traffic": {
      "direction": "outbound"
    }
  }
}

O que isso produz (se direção é "unknown")​

{
  "normalized": {}
}

Por que funciona​

when: { not: { ... } }: O predicado not inverte a lógica do seu filho. O filho verifica equals, que é verdadeiro apenas se direction == "unknown". O not inverte, então a regra executa apenas se a direção NÃO é "unknown".

Variações / pegadinhas​

  • Lógica aninhada: not pode envolver qualquer outro predicado. not: { in: { ... } } significa "valor NÃO está na lista."
  • Nenhum not aninhado: not: { not: { ... } } é permitido mas desajeitado; apenas use o predicado base.

Receita 8: Transformação de Tática MITRE​

Problema: Sophos envia táticas MITRE como [{"tactic": {"id": "TA0001", "name": "Initial Access"}}, ...], OCSF quer [{"tactics": [{"uid": "TA0001", "name": "Initial Access"}], "version": "16.1"}, ...].

Contexto do vendor: Detecção do Sophos XDR (com táticas MITRE).

Evento de entrada (recorte)​

{
  "id": "det_004",
  "mitreAttacks": [
    { "tactic": { "id": "TA0001", "name": "Initial Access" } },
    { "tactic": { "id": "TA0002", "name": "Execution" } }
  ]
}

Regra(s) de mapeamento​

{
  "target": "normalized.attacks",
  "source": "mitreAttacks",
  "type_cast": "mitre_tactic_to_ocsf"
}

O que isso produz​

{
  "normalized": {
    "attacks": [
      { "tactics": [{ "uid": "TA0001", "name": "Initial Access" }], "version": "16.1" },
      { "tactics": [{ "uid": "TA0002", "name": "Execution" }], "version": "16.1" }
    ]
  }
}

Por que funciona​

type_cast: "mitre_tactic_to_ocsf": Este cast especializado sabe como transformar a forma do Sophos em forma OCSF. Ele:

  1. Valida que cada item da lista tem um dicionário tactic com id e name.
  2. Renomeia id → uid.
  3. Envolve a tática em uma lista tactics.
  4. Adiciona versão OCSF "16.1".
  5. Silenciosamente filtra táticas null (Sophos às vezes as emite em alertas de baixa fidelidade).

Variações / pegadinhas​

  • Sem táticas: Se mitreAttacks é null ou vazio, saída é null ou [].
  • Táticas malformadas: Se um item tático falta id ou name, um OperatorError é lançado e o evento é quarentenado.
  • Táticas null na lista: Itens { "tactic": null } são silenciosamente pulados; não é um erro.

Receita 9: Normalização de Score​

Problema: Vendor envia alertScore: 0.75 (float 0-1), OCSF espera int 0-100.

Contexto do vendor: Vendor A envia probabilidade/confiança como float.

Evento de entrada (recorte)​

{
  "id": "alert_001",
  "alertScore": 0.85
}

Regra(s) de mapeamento​

{
  "target": "normalized.confidence",
  "source": "alertScore",
  "type_cast": "score_to_percent"
}

O que isso produz​

{
  "normalized": {
    "confidence": 85
  }
}

Por que funciona​

type_cast: "score_to_percent": Multiplica por 100 e arredonda. 0.85 × 100 = 85. Trata:

  • Entrada já em [0, 100]? Passthrough idempotente.
  • Entrada None? Retorna None.
  • Entrada fora do intervalo? Lança OperatorError (quarentena o evento).
  • Entrada é bool? Lança OperatorError (bool é tecnicamente um int em Python; rejeitado para pegar bugs).

Variações / pegadinhas​

  • Fixação: Se vendor envia 1.1 (fora do intervalo [0, 1]), cast falha. Você deve value_map ou pre_cast para corrigir primeiro.
  • Integer já 0-100: Se vendor envia int 85, cast verifica intervalo e passa inalterado.

Receita 10: Campo Obrigatório com Default e Validação​

Problema: time é obrigatório em OCSF. Se o vendor não o fornece, use um timestamp de fallback. Se mesmo isso falha, quarentene o evento.

Contexto do vendor: A maioria dos vendors; alguns têm gaps de timestamp.

Evento de entrada (recorte)​

{
  "id": "alert_001",
  "createdAt": "2026-04-27T12:30:00Z"
}

Regra(s) de mapeamento​

{
  "target": "normalized.time",
  "source": "createdAt",
  "fallback_source": ["raisedAt", "processedAt"],
  "default": 0,
  "type_cast": "iso_to_epoch",
  "required": true
}

O que isso produz (caminho feliz)​

{
  "normalized": {
    "time": 1745764200
  }
}

O que isso produz (todos timestamps ausentes, default usado)​

{
  "normalized": {
    "time": 0
  }
}

Por que funciona​

Ordem de execução:

  1. Tentar createdAt → se null, tentar raisedAt.
  2. Se ainda null, tentar processedAt.
  3. Se ainda null, usar default: 0 (epoch zero).
  4. Aplicar type_cast: iso_to_epoch. Se o valor é 0 (um int), cast passa idempotente (ints de epoch ignoram análise).
  5. Verificar required: true — valor final não é null, então sem quarentena.

Quando falha: Se você removeu default: 0 e todos os timestamps estão ausentes, valor final seria null, required: true seria acionado, e evento é quarentenado com error_kind: "map".

Variações / pegadinhas​

  • Sempre usar default 0 para time? Não — epoch 0 é 1970-01-01T00:00:00Z, que é confuso. Use default: null e deixe required: true quarentene eventos com time ausente.
  • Prefira timestamp recente como default: Use timestamp Unix de hora de criação de mapeamento ou UTC agora, mas documente no comentário do mapeamento.

Receita 11: Dedup de Array​

Problema: Lista envelopeRecipients tem duplicatas: ["user@domain.com", "user@domain.com", "admin@domain.com"]. Você quer apenas endereços únicos.

Contexto do vendor: Email do Sophos (raro mas acontece com regras de encaminhamento).

Evento de entrada (recorte)​

{
  "id": "det_005",
  "envelopeRecipients": ["user@domain.com", "user@domain.com", "admin@domain.com"]
}

Regra(s) de mapeamento — via array_builder dedup_by​

{
  "target": "normalized.observables",
  "kind": "array_builder",
  "items": [
    {
      "name": "email_to",
      "type": "Email Address",
      "type_id": 5,
      "source": "envelopeRecipients",
      "explode": true
    }
  ],
  "dedup_by": ["value"]
}

O que isso produz​

{
  "normalized": {
    "observables": [
      { "name": "email_to", "type": "Email Address", "type_id": 5, "value": "user@domain.com" },
      { "name": "email_to", "type": "Email Address", "type_id": 5, "value": "admin@domain.com" }
    ]
  }
}

Por que funciona​

array_builder dedup_by: Depois de construir a lista de observáveis, dedup_by: ["value"] remove observáveis duplicados que têm o mesmo campo value. A primeira ocorrência é mantida.

Timing de dedup: Aplicado depois que todos os itens são coletados e explodidos.

Alternativa — via type_cast dedup (para listas simples)​

{
  "target": "normalized.email_recipients",
  "source": "envelopeRecipients",
  "type_cast": "dedup"
}

Isso produz:

{
  "normalized": {
    "email_recipients": ["user@domain.com", "admin@domain.com"]
  }
}

Ambas as abordagens funcionam; escolha baseado se você precisa de observáveis (array_builder) ou uma lista plana (type_cast dedup).

Variações / pegadinhas​

  • dedup_by múltiplos campos: dedup_by: ["type_id", "value"] deduplica em tipo e valor.
  • Itens não hash: Se observáveis contêm dicionários aninhados, dedup os pula (não verdadeiramente deduplicado) para evitar crashes.

Receita 12: Marcar Regra de Espaço Reservado Intencional​

Problema: Vendor A não fornece IP de origem em seus alertas, mas a classe OCSF Detection Finding a recomenda. Você adiciona uma regra com default null e quer que o dry-run não a sinalize como um bug.

Contexto do vendor: Vendor A (exemplo genérico).

Evento de entrada (recorte)​

{
  "id": "alert_001",
  "description": "Policy violation",
  "severity": "medium"
}

Regra(s) de mapeamento​

{
  "target": "normalized.src_endpoint.ip",
  "source": "sourceIp",
  "default": null,
  "expected_always_default": true
}

O que isso produz​

{
  "normalized": {
    "src_endpoint": {
      "ip": null
    }
  }
}

Por que funciona​

expected_always_default: true: Diz à ferramenta dry-run "Eu sei que esta regra sempre usa o default; não avise sobre isso." Quando dry-run escaneia seu mapeamento:

  1. Encontra que sourceIp nunca está nos eventos do vendor.
  2. Regra sempre resolve para default: null.
  3. Normalmente, dry-run a sinaliza como "regra sempre usa default — possível bug de mapeamento?"
  4. Mas expected_always_default: true suprime o aviso.

Semântica: Flag NÃO muda a execução; a regra ainda executa e escreve null. É puramente um sinal para diagnósticos.

Variações / pegadinhas​

  • Por que não omitir a regra? Se você acha que o vendor PODE adicionar o campo depois (ex: "feature planejada para próxima versão"), mantenha a regra com este flag como seguro.
  • Uso excessivo: Se cada terceira regra tem este flag, seu mapeamento é incompleto. Use com moderação para gaps conhecidos.
  • Documentação: Adicione um comentário à regra explicando por que sempre é default.
