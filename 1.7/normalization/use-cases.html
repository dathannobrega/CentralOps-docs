Casos de Uso — Guias Passo a Passo

Três integrações de vendor reais, mostrando como criar e depurar um mapeamento completo do zero.

Caso de Uso 1: Detecção do Sophos XDR (Email)​

Vendor: Evento de Detecção do Sophos
Tipo de evento: Detecção de segurança de email
Complexidade: Alta (pré-processamento de JSON aninhado, múltiplos observáveis)

Evento do Vendor (Entrada)​

Sophos envia uma detecção como JSON plano com um campo processedData contendo JSON stringificado.

{
  "id": "det_20260427_001",
  "time": "2026-04-27T14:30:00Z",
  "severity": 7,
  "detectionRule": "Email.Phishing.AdvancedGraphics",
  "attackType": "Phishing",
  "type": ["Email"],
  "mitreAttacks": [
    {
      "tactic": {
        "id": "TA0001",
        "name": "Initial Access"
      }
    }
  ],
  "device": {
    "entity": "john-desktop",
    "id": "device_123",
    "type": "linux"
  },
  "processedData": "{\"parsedAlert\":{\"fields\":{\"clientIp\":\"192.0.2.15\",\"mailFrom\":\"attacker@evil.com\",\"envelopeRecipients\":[\"john@company.com\",\"admin@company.com\"],\"attachments\":[{\"name\":\"invoice_2026.exe\",\"checksum\":\"d41d8cd98f00b204e9800998ecf8427e\"},{\"name\":\"document.pdf\",\"checksum\":\"356a192b7913b04c54574d18c28d46e6\"}]}}}",
  "rawData": {
    "raw": "{\"header\":{\"from\":\"attacker@evil.com\"}}"
  }
}

Regras de Mapeamento​

{
  "preprocess": [
    {
      "op": "json_parse",
      "source": "processedData",
      "target": "_processed",
      "tolerant": true
    },
    {
      "op": "json_parse",
      "source": "rawData.raw",
      "target": "_raw_email",
      "tolerant": true
    }
  ],
  "rules": [
    // ── Classe OCSF (obrigatória) ──
    { "target": "normalized.class_uid", "const": 2004, "required": true },
    { "target": "normalized.class_name", "const": "Detection Finding" },
    { "target": "normalized.category_uid", "const": 2, "required": true },
    { "target": "normalized.category_name", "const": "Findings" },
    { "target": "normalized.activity_id", "const": 1 },
    { "target": "normalized.activity_name", "const": "Create" },
    { "target": "normalized.type_uid", "const": 200401 },

    // ── Timing (obrigatório) ──
    {
      "target": "normalized.time",
      "source": "time",
      "type_cast": "iso_to_epoch",
      "required": true
    },

    // ── Identificação de Finding ──
    {
      "target": "normalized.finding_info.uid",
      "source": "id",
      "required": true
    },
    {
      "target": "normalized.finding_info.title",
      "source": "detectionRule",
      "fallback_source": ["attackType"],
      "default": null
    },
    {
      "target": "normalized.finding_info.types",
      "source": "[type]",
      "default": []
    },
    {
      "target": "normalized.finding_info.created_time",
      "source": "time",
      "type_cast": "iso_to_epoch"
    },

    // ── Severidade ──
    {
      "target": "normalized.severity_id",
      "source": "severity",
      "type_cast": "to_int",
      "default": 0,
      "required": true
    },
    {
      "target": "normalized.severity",
      "source": "severity",
      "type_cast": "to_str",
      "default": "Unknown"
    },

    // ── Status ──
    { "target": "normalized.status_id", "const": 1 },
    { "target": "normalized.status", "const": "New" },

    // ── Mensagem ──
    {
      "target": "normalized.message",
      "source": "detectionRule",
      "default": null
    },

    // ── Dispositivo ──
    {
      "target": "normalized.device.name",
      "source": "device.entity",
      "default": null
    },
    {
      "target": "normalized.device.uid",
      "source": "device.id",
      "default": null
    },
    {
      "target": "normalized.device.type",
      "source": "device.type",
      "default": null
    },

    // ── Táticas MITRE ──
    {
      "target": "normalized.attacks",
      "source": "mitreAttacks",
      "type_cast": "mitre_tactic_to_ocsf",
      "default": []
    },

    // ── Metadados ──
    { "target": "normalized.metadata.version", "const": "1.3.0" },
    { "target": "normalized.metadata.product.name", "const": "Sophos Central" },
    { "target": "normalized.metadata.product.vendor_name", "const": "Sophos" },
    { "target": "normalized.metadata.product.feature.name", "const": "XDR Detections" },
    {
      "target": "normalized.metadata.uid",
      "source": "id"
    },

    // ── Observáveis (array_builder: IPs, emails, file hashes) ──
    {
      "target": "normalized.observables",
      "kind": "array_builder",
      "items": [
        {
          "name": "src_ip",
          "type": "IP Address",
          "type_id": 2,
          "source": "_processed.parsedAlert.fields.clientIp",
          "skip_null": true
        },
        {
          "name": "email_from",
          "type": "Email Address",
          "type_id": 5,
          "source": "_processed.parsedAlert.fields.mailFrom",
          "skip_null": true
        },
        {
          "name": "email_to",
          "type": "Email Address",
          "type_id": 5,
          "source": "_processed.parsedAlert.fields.envelopeRecipients",
          "explode": true,
          "skip_null": true
        },
        {
          "name": "file_hash",
          "type": "Hash",
          "type_id": 8,
          "source": "_processed.parsedAlert.fields.attachments[*].checksum",
          "explode": true,
          "skip_null": true
        },
        {
          "name": "file_name",
          "type": "File Name",
          "type_id": 7,
          "source": "_processed.parsedAlert.fields.attachments[*].name",
          "explode": true,
          "skip_null": true
        }
      ],
      "skip_null": true,
      "dedup_by": ["value"]
    }
  ]
}

Explicação Regra-por-Regra​

  1. Pré-processamento: json_parse em processedData → _processed. Regras subsequentes acessam detalhes aninhados de email via _processed.parsedAlert.fields.*.

  2. Classe/Categoria/Atividade: Todas constantes OCSF. Detection Finding = 2004, categoria Findings = 2, atividade Create = 1.

  3. Tempo: Obrigatório. source: "time" + iso_to_epoch converte ISO-8601 para Unix epoch. Se ausente, evento quarentenado.

  4. UID do Finding & título: Source primária detectionRule é o nome da regra. Se null, fallback para attackType. Ambos os campos presentes em detecção do Sophos, então fallback é seguro para casos extremos.

  5. Severidade: Int do vendor (escala 0-10). Armazenado como está em severity_id. default: 0 se ausente. Nota: Escala do vendor difere de OCSF 1-6; se você precisa redimensionar, adicione value_map.

  6. Dispositivo: Extrair informações de dispositivo de objeto aninhado device.*. Defaults null permitem informações de dispositivo ausentes.

  7. Observáveis:

    • src_ip: IP único de clientIp pré-processado.
    • email_from: Endereço de remetente único.
    • email_to: Múltiplos destinatários — explode: true cria um observável por elemento.
    • file_hash & file_name: Array de anexos. JMESPath attachments[*].checksum extrai todos os checksums; combinado com explode: true, produz um observável por hash.
    • dedup_by: ["value"]: Remover observáveis duplicados (ex: se lista de destinatários tem duplicatas).

Ao Criar Este Mapeamento​

  1. Comece com constantes (classe, categoria, atividade) — não mudam por vendor.
  2. Adicione campos obrigatórios (time, UID do finding) — falhe rápido se ausentes.
  3. Adicione severidade — mais importante para filtragem/roteamento.
  4. Adicione informações de dispositivo & ator — contexto para investigação.
  5. Adicione observáveis — extrair IPs, emails, hashes. Use array_builder e explode para listas.
  6. Use dry-run para verificar cobertura. Se algum campo de vendor não é usado, Drift Explorer o sinalizará.

Tipo de evento: Alerta de firewall ou EDR

Diferenças do Mapeamento de Detecção​

  1. Sem pré-processamento: Dados de alerta são planos; sem necessidade de analisar JSON aninhado.
  2. Fonte de tempo: Usa JMESPath || (fallback dentro da expressão) em vez do campo DSL fallback_source. Ambos funcionam; fallback_source é mais claro para múltiplas opções.
  3. Busca de severidade: String-to-int value_map em vez de int direto. Alertas do Sophos enviam severidade como texto.
  4. Observáveis mais simples: UID de dispositivo, nome, ID de tenant. Sem listas explodidas ou caminhos complexos.
  5. Sem táticas MITRE: Alertas não incluem táticas; removidas deste mapeamento.

Caso de Uso 3: Acme EDR (Scaffold de Template)​

Vendor: Acme EDR (fictício)
Tipo de evento: Detecção/Ameaça
Complexidade: Scaffold inicial para novo vendor

Cenário​

Você está integrando "Acme EDR", uma nova plataforma de detecção de endpoint. Você tem 10 eventos de exemplo. Você precisa criar um mapeamento em 30 minutos.

Passo 1: Escolha Template OCSF​

Revise a classe OCSF Detection Finding. Campos obrigatórios comuns:

- class_uid (2004)
- category_uid (2)
- activity_id (1)
- time (unix epoch)
- finding_info.uid (ID único)
- severity_id (1-6)
- metadata.version
- metadata.product.*

Passo 2: Mapear Campos de Vendor​

Escaneie amostras de evento Acme para equivalentes:

{
  "event_id": "evt_001",
  "timestamp": 1746014700,
  "alert_name": "Suspicious Process",
  "alert_severity": 4,
  "machine": {
    "hostname": "workstation-42",
    "os": "windows"
  },
  "user": {
    "username": "alice.smith"
  },
  "process": {
    "name": "cmd.exe",
    "parent": "explorer.exe"
  },
  "file": {
    "path": "C:\\Temp\\payload.exe",
    "hash": "abc123"
  }
}

Mapping:

OCSF FieldAcme FieldNotes
timetimestampAlready epoch int
finding_info.uidevent_idUnique detection ID
finding_info.titlealert_nameWhat triggered the alert
severity_idalert_severityInteger 1-5; OCSF wants 1-6 → use default 0 if absent
device.namemachine.hostnameEndpoint name
device.typemachine.osPlatform
actor.user.nameuser.usernameWho was logged in
process.*process.*Suspicious process details
observablesprocess.name, file.path, file.hashIOCs

Step 3: Write Initial Mapping​

{
  "rules": [
    // ── OCSF constants ──
    { "target": "normalized.class_uid", "const": 2004, "required": true },
    { "target": "normalized.class_name", "const": "Detection Finding" },
    { "target": "normalized.category_uid", "const": 2, "required": true },
    { "target": "normalized.category_name", "const": "Findings" },
    { "target": "normalized.activity_id", "const": 1 },
    { "target": "normalized.activity_name", "const": "Create" },
    { "target": "normalized.type_uid", "const": 200401 },

    // ── Timing ──
    { "target": "normalized.time", "source": "timestamp", "required": true },

    // ── Finding ──
    { "target": "normalized.finding_info.uid", "source": "event_id", "required": true },
    { "target": "normalized.finding_info.title", "source": "alert_name", "default": null },

    // ── Severity ──
    {
      "target": "normalized.severity_id",
      "source": "alert_severity",
      "type_cast": "to_int",
      "default": 0,
      "required": true
    },

    // ── Device ──
    { "target": "normalized.device.name", "source": "machine.hostname", "default": null },
    { "target": "normalized.device.type", "source": "machine.os", "default": null },

    // ── Actor ──
    { "target": "normalized.actor.user.name", "source": "user.username", "default": null },

    // ── Metadata ──
    { "target": "normalized.metadata.version", "const": "1.0.0" },
    { "target": "normalized.metadata.product.name", "const": "Acme EDR" },
    { "target": "normalized.metadata.product.vendor_name", "const": "Acme" },
    { "target": "normalized.metadata.uid", "source": "event_id" },

    // ── Observables ──
    {
      "target": "normalized.observables",
      "kind": "array_builder",
      "items": [
        { "name": "process_name", "type": "Process Name", "type_id": 29, "source": "process.name", "skip_null": true },
        { "name": "file_path", "type": "File Name", "type_id": 7, "source": "file.path", "skip_null": true },
        { "name": "file_hash", "type": "Hash", "type_id": 8, "source": "file.hash", "skip_null": true }
      ],
      "skip_null": true,
      "dedup_by": ["value"]
    }
  ]
}

Passo 4: Testar com Dry-Run​

  1. Salve o mapping na UI do CentralOps.
  2. Clique em Dry Run com seus 10 eventos de amostra.
  3. Verifique a saída:
    • Todos os eventos passaram? Procure avisos de "100% default" → preencha lacunas.
    • Algum evento foi para quarentena? Verifique error_kind — provavelmente um campo required ausente ou falha de type_cast.
    • Algum campo do vendor não usado? O Drift Explorer lista — decida se é relevante (ignore IDs opacos, adicione campos faltantes).

Passo 5: Iterar​

Exemplos de achados do dry-run:

Achado 1: machine.network_interfaces[*].ip está presente em 50% dos eventos, não usado no mapping.
Ação: Adicionar aos observables:

{ "name": "device_ip", "type": "IP Address", "type_id": 2, "source": "machine.network_interfaces[*].ip", "explode": true, "skip_null": true }

Achado 2: alert_severity às vezes null; regra cai no default 0.
Ação: Marcar regra como intencional:

"expected_always_default": true

Achado 3: process.parent não usado. Time da Acme confirma que não é acionável.
Ação: Ignorar (não adicionar ao mapping).

Passo 6: Subir​

Quando o dry-run mostrar cobertura razoável (70%+ dos campos mapeados ou intencionalmente ignorados), salve e ative.

Resumo do fluxo​

FaseAçãoEsforço
DescobertaRevisar docs do vendor e eventos de amostra5 min
ScaffoldEscrever constantes OCSF + campos obrigatórios5 min
MappingAdicionar regras vendor → OCSF10 min
TestesDry-run, revisar drift, iterar10 min
Total~30 minutos

Dicas:

  • Comece pelos campos required para falhar rápido.
  • Use fallback_source para campos que podem aparecer em múltiplos campos do vendor.
  • Use array_builder + explode para lidar com listas de tamanho variável.
  • Rode dry-run cedo e com frequência — captura 80% dos problemas antes de produção.
