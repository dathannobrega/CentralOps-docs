CML — CentralOps Mapping Language

CML é a linguagem declarativa do CentralOps para normalizar eventos brutos de vendors de segurança em OCSF. Esta página explica o que CML é como linguagem — princípios, modelo de execução, governança e diferenciação. Para a referência formal de cada campo, veja Especificação DSL.

O que é CML​

CML é uma DSL JSON declarativa especializada em normalização vendor → OCSF. Você descreve, regra por regra:

  • De onde vem o valor no payload do vendor (source JMESPath, const literal, ou cadeia de fallback_source)
  • Para onde ele vai no envelope normalizado (target dot-path em normalized.*)
  • Como ele é transformado (value_map, pre_cast, type_cast)
  • Quando a regra deve executar (when com predicates: exists, equals, in, not)
  • O que acontece se faltar (default, required — o último manda o evento pra quarentena)

Cada mapeamento é um documento JSON com duas seções: preprocess (extrações reutilizáveis, executadas uma vez) e rules (a sequência de transformações campo a campo).

CML não é:

  • Uma pipeline language genérica (não é Tenzir TQL, não é Cribl SPL, não é Vector VRL)
  • Uma linguagem imperativa (sem variáveis mutáveis, sem if/else arbitrário, sem loops)
  • Uma DSL Turing-completa (intencional — o que ela perde em expressividade ganha em previsibilidade e auditoria)

CML é:

  • Declarativa — você descreve o quê, não o como
  • Domain-specific — primitivas pensadas pra normalização, não pra streaming genérico
  • Versionada nativamente — cada commit vira MappingVersion append-only no banco
  • Auditável — diff entre versões, dry-run contra sample reservoir, audit trail por usuário
  • Drift-aware — campos novos do vendor são detectados e surfaceados automaticamente

Por que CML existe (princípios de design)​

A categoria "security data pipeline" tem várias linguagens estabelecidas. Por que CML em vez de adotar uma delas?

LinguagemCategoriaPor que não é CML
Tenzir TQLDataflow streaming geralExcelente pra pipeline genérico (filtros, joins, agregações); over-engineered pra mapping field-by-field
Cribl SPLSplunk-inspired routingBoa pra rotear/transformar logs; sem governança nativa (versionamento, dry-run, rollback)
Vector VRLDatadog/Mezmo transformerExpressiva, mas imperativa e sem audit trail; cliente precisa de Git workflow externo
JavaScript / Python customCódigo proceduralMáxima flexibilidade, mínima auditoria; impossível dry-run determinístico

CML escolhe menos expressividade, mais governança. Princípios que governam o design:

1. Declarativa, não imperativa​

Uma regra CML é uma fotografia: "este campo do vendor vira este campo OCSF, com estas transformações". Não há fluxo de controle. Isso permite:

  • Compilação determinística — o mesmo mapping sempre gera a mesma AST
  • Diff semântico — diff entre versões mostra mudança de comportamento, não mudança de linhas de código
  • Dry-run reprodutível — rodar contra sample reservoir produz resultado idêntico em qualquer momento

2. Especializada, não geral​

CML não tem for, if/else arbitrário, joins entre streams, agregações, ou window functions. Tem when (guarda condicional de uma regra), fallback_source (cadeia de tentativa), value_map (lookup), array_builder (observables OCSF). Cada primitiva nasceu de um caso de uso real de normalização.

3. Governança nativa​

Versionamento, dry-run, rollback e drift detection não são features adicionais — são parte do runtime. Você não consegue commitar um mapping novo sem ter rodado dry-run e recebido um ack_token válido. Você não consegue perder uma versão antiga — todas vivem em mapping_versions append-only.

4. JMESPath dentro, não Turing-completo fora​

Pra extrair valor do payload bruto, CML usa JMESPath — uma query language consolidada e largamente conhecida. Você ganha pipes, filtros, funções, mas dentro do escopo de "extrair valor de um JSON". Não precisa aprender uma nova linguagem de expressão.

Modelo de execução​

Cada evento bruto passa por uma sequência fixa, sem ramificações:

┌──────────────────────────────────────────────────────────────────────┐
│  raw payload (vendor)                                                │
└──────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────────────────┐
│  preprocess[]   (uma vez por evento, populando _processed)           │
│    op: json_parse → extrai e parseia JSON aninhado                   │
└──────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────────────────┐
│  rules[] (uma a uma, em ordem)                                       │
│                                                                       │
│   Para cada regra:                                                    │
│     1. when?  ──no──▶ pula regra (target não é escrito)              │
│        │ yes                                                          │
│     2. source / const / fallback_source → valor                       │
│     3. default (se valor null)                                        │
│     4. pre_cast (operador, antes do lookup)                           │
│     5. value_map (lookup case-insensitive)                            │
│     6. type_cast (operador, final)                                    │
│     7. required? null → quarentena (error_kind=map)                   │
│     8. escreve em target (dot-path)                                   │
└──────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────────────────┐
│  envelope canônico                                                    │
│    { _centralops: {...lineage...},                                    │
│      normalized: {...output OCSF...},                                 │
│      raw: {...payload preservado...} }                                │
└──────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
                     Dispatcher pluggable
                  (Wazuh / Splunk / Elastic / ...)

Pontos importantes do modelo:

  • Ordem garantida — default → pre_cast → value_map → type_cast. Determinística.
  • Falha = quarentena — required: true + valor null produz MappingRequiredFieldError, evento vai pra quarantine_events com error_kind="map". Nunca silenciosamente descartado.
  • Drift = surface — paths do raw não consumidos por nenhuma regra são detectados pela drift detection e surfaceados na UI pra triage.
  • Raw preservado — o payload original vai pro envelope sob raw.*. Auditoria total.

Sintaxe em 30 segundos​

Mapping mínimo funcional:

{
  "preprocess": [],
  "rules": [
    // metadados OCSF fixos
    { "target": "normalized.class_uid",   "const": 2004 },
    { "target": "normalized.class_name",  "const": "Detection Finding" },

    // campo simples com type cast
    {
      "target": "normalized.time",
      "source": "eventTime",
      "type_cast": "iso_to_epoch",
      "required": true
    },

    // severity com lookup
    {
      "target": "normalized.severity_id",
      "source": "severity",
      "value_map": { "critical": 5, "high": 4, "medium": 3, "low": 2 },
      "default": 0
    },

    // título com fallback em cadeia
    {
      "target": "normalized.finding_info.title",
      "source": "ruleTitle",
      "fallback_source": ["attackType", "description"],
      "default": "Unknown Detection"
    },

    // regra condicional
    {
      "target": "normalized.actor.user.email_addr",
      "source": "mailFrom",
      "when": { "exists": "mailFrom" }
    },

    // observables OCSF a partir de várias sources
    {
      "target": "normalized.observables",
      "kind": "array_builder",
      "items": [
        { "name": "src_ip", "type": "IP Address", "type_id": 2, "source": "remoteIp" },
        { "name": "email_to", "type": "Email Address", "type_id": 5, "source": "recipients", "explode": true }
      ],
      "skip_null": true,
      "dedup_by": ["value"]
    }
  ]
}

Governança nativa​

CML não é "DSL e Git" — o ciclo de vida vive dentro do produto.

Versionamento append-only​

Toda mudança em um mapping cria uma MappingVersion no banco. A versão anterior nunca é deletada. MappingDefinition.current_version_id aponta pra qual está ativa. Rollback é flip de ponteiro.

MappingDefinition: "sophos.detections"
  ├─ MappingVersion #1 (2026-01-12, autor: alice)        ← deprecated
  ├─ MappingVersion #2 (2026-02-08, autor: bob)          ← deprecated
  └─ MappingVersion #3 (2026-03-15, autor: carol) ★ ACTIVE

Dry-run obrigatório antes de commit​

Você não consegue publicar um mapping novo direto. O fluxo é:

  1. POST /mappings/dry-run — rode o mapping contra o sample reservoir (eventos brutos coletados nos últimos dias, preservados pra esse fim)
  2. O backend retorna estatísticas: % de regras que aplicaram, % de quarentena previsto, campos sempre nulos (always_default), drift novo introduzido
  3. Você recebe um ack_token assinado, com TTL curto, atrelado ao hash do mapping
  4. POST /mappings/{id}/versions — submete o mapping com o ack_token. Backend valida hash e expira o token

Resultado: mapping em produção sempre passou por dry-run em sample reservoir. Sem ack_token válido → commit recusado.

Audit trail​

Cada operação (criar versão, rollback, commit) registra:

  • Usuário (author_user_id)
  • Timestamp
  • commit_message (livre, recomendado obrigatório por policy)
  • diff_from_previous (computado automaticamente: regras adicionadas/removidas/alteradas)
  • dry_run_stats (snapshot do dry-run que validou esta versão)

Exportável em formato de conformidade.

Drift detection​

A cada coleta, o engine compara os paths consumidos pelas regras com os paths presentes no raw payload. Diferenças (paths que apareceram mas ninguém mapeou) viram entradas em unknown_fields. A UI Drift Explorer permite:

  • Filtrar por vendor / event_type
  • Marcar campos como "ignorar" (não vão mais alertar)
  • Marcar campos como "mapear" (entram no backlog do engineer)
  • Bulk-ignore quando vendor faz update e adiciona telemetria interna irrelevante

Drift é silencioso em pipelines genéricos. Em CML é surface obrigatória.

Rollback em um clique​

POST /mappings/{id}/rollback?to_version=<version_id>

Atomicidade: current_version_id é atualizado em transação. Nenhum evento em vôo é perdido — o engine recompila o mapping na próxima execução de task, em até segundos.

Como CML se compara a outras linguagens​

A tabela abaixo é honesta — CML perde em expressividade transformacional pura. Ganha em governança nativa e foco no caso de uso de normalização.

CapacidadeCMLTenzir TQLCribl SPLVector VRL
CategoriaMapping languagePipeline languageRouting/transformTransform language
ParadigmaDeclarativaDataflow funcionalImperativa pipe-styleImperativa scripting
Field-by-field mapping★ EspecializadaAdequadaAdequadaAdequada
Filtros / drops condicionaisLimitada (only when por regra)★ Forte★ Forte★ Forte
Roteamento multi-destinoNão (no pipeline, não na linguagem)★ Sim★ Sim★ Sim
Agregações / joinsNão★ SimLimitadaLimitada
Versionamento nativo★ Sim (append-only no banco)Não (Git externo)Não (deploy snapshot)Não (Git externo)
Dry-run com sample reservoir★ Sim (obrigatório)NãoManualManual
Rollback append-only★ SimNãoNãoNão
Drift detection★ Sim (UI dedicada)NãoNãoNão
Audit trail por mudança★ Sim (autor + diff + dry-run snapshot)NãoNãoNão
Curva de aprendizadoBaixa (JSON + JMESPath)Média (linguagem nova)Média (Splunk-inspired)Média-alta (scripting)

Se você precisa de: streaming geral, agregações, multi-destino dinâmico — TQL ou Vector são melhores. Se você precisa de: normalização auditável com governança nativa e foco MSSP — CML é o que você quer.

Exemplo end-to-end​

Entrada: detecção bruta do Sophos Central:

{
  "id": "evt-abc-123",
  "createdAt": "2026-05-28T14:32:18Z",
  "severity": "high",
  "ruleName": "PowerShell.Suspicious",
  "endpoint": {
    "id": "ep-007",
    "hostname": "WORKSTATION-7",
    "os": "Windows 11"
  },
  "processedData": "{\"parsedAlert\":{\"fields\":{\"clientIp\":\"10.20.30.40\"}}}"
}

Mapping CML:

{
  "preprocess": [
    { "op": "json_parse", "source": "processedData", "target": "_processed", "tolerant": true }
  ],
  "rules": [
    { "target": "normalized.class_uid",  "const": 2004 },
    { "target": "normalized.class_name", "const": "Detection Finding" },
    { "target": "normalized.time",       "source": "createdAt", "type_cast": "iso_to_epoch", "required": true },
    { "target": "normalized.severity_id",
      "source": "severity",
      "value_map": { "critical": 5, "high": 4, "medium": 3, "low": 2 },
      "default": 0 },
    { "target": "normalized.finding_info.title",
      "source": "ruleName",
      "default": "Unknown Detection" },
    { "target": "normalized.finding_info.uid",
      "source": "id",
      "required": true },
    { "target": "normalized.device.name",
      "source": "endpoint.hostname" },
    { "target": "normalized.device.os.name",
      "source": "endpoint.os" },
    { "target": "normalized.src_endpoint.ip",
      "source": "_processed.parsedAlert.fields.clientIp",
      "when": { "exists": "_processed.parsedAlert.fields.clientIp" } }
  ]
}

Envelope canônico produzido:

{
  "_centralops": {
    "schema_version": "1.0",
    "ocsf_version": "1.4.0",
    "vendor": "sophos",
    "integration_id": "int-42",
    "customer_id": "iris-cust-100",
    "stream": "detections",
    "event_type": "sophos.detections",
    "mapping_version_id": "mv-2026-03-15-carol",
    "collected_at": "2026-05-28T14:33:02Z",
    "event_id": "evt-abc-123"
  },
  "normalized": {
    "class_uid": 2004,
    "class_name": "Detection Finding",
    "time": 1748441538,
    "severity_id": 4,
    "finding_info": { "title": "PowerShell.Suspicious", "uid": "evt-abc-123" },
    "device": { "name": "WORKSTATION-7", "os": { "name": "Windows 11" } },
    "src_endpoint": { "ip": "10.20.30.40" }
  },
  "raw": { /* payload original preservado */ }
}

O envelope segue pra qualquer dispatcher pluggable (Wazuh hoje, outros via _Target Protocol).
