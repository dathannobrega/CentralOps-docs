Secrets e Master Key

Criptografia de credenciais em repouso e rotação de chaves mestras.

Público: Admin only.

O que é criptografado​

Campos sensíveis de cada integração:

CampoExemplo
client_id"abc123xyz" (Sophos Client ID)
client_secret"secret_key_value"
api_keyToken de API
access_tokenToken OAuth 2.0
refresh_tokenToken de renovação
api_passwordSenha (Wazuh)
indexer_passwordSenha (Wazuh Indexer)

Algoritmo de criptografia​

  • Cipher: AES-256-GCM (Advanced Encryption Standard, chave 256 bits, Galois/Counter Mode).
  • Key derivation: PBKDF2 com 100k iterações, aplicado a APP_MASTER_KEY.
  • Nonce: gerado aleatoriamente por secret (IV único).
  • Autenticação: tag GCM verifica integridade (tampering detectado).

Resultado: cada secret é um blob criptografado que:

  • Não pode ser lido sem APP_MASTER_KEY.
  • Detecta tampering (tentativa de editar cipher text será rejeitada).
  • É específico (mesmo secret em outra integração terá nonce diferente).

APP_MASTER_KEY​

O que é: chave raiz que deriva todas as chaves de criptografia.

Geração (primeira vez):

Ao iniciar container sem APP_MASTER_KEY em .env:

Sistema gera:  256 bits aleatórios (32 bytes = 64 caracteres hex)
Armazena em:   /app/data/app_master_key (volume persistente)
Log:           "Master key generated and persisted"

Em produção: defina em .env para não depender de geração automática:

# Gerar localmente
APP_MASTER_KEY=$(openssl rand -hex 32)
echo "APP_MASTER_KEY=$APP_MASTER_KEY" >> .env

Segurança: trate como senha root. Guarde em cofre seguro (Vault, LastPass, etc).

Rotação de master key​

Por quê rotar:

  • Auditoria anual (compliance).
  • Suspeita de comprometimento.
  • Mudança de acesso restrito.
  • Saída de admin (novo admin não conhece antiga key).

Processo (via UI):

  1. Vá a Administração → Configuração.
  2. Clique Rotar master key.
  3. Confirmação: "Sistema re-cifra todos os secrets (~5 min)".
  4. Clique Confirmar.

O que acontece:

1. Sistema gera nova chave (256 bits)
2. Decifra todos os secrets com chave antiga
3. Re-cifra todos os secrets com nova chave
4. Armazena nova chave em /app/data/app_master_key
5. Backup da chave antiga em /app/data/app_master_key.backup
6. Registra em audit log: "Master key rotated by admin-user"
7. Restart automático se necessário

Não há downtime: operação é atômica.

Sem interrupção de coleta: integrations continuam coletando durante rotação.

Backup e recovery​

Backup (recomendado)​

Após rotar key, faça backup:

# Dentro do container
docker compose exec centralops cat /app/data/app_master_key > /tmp/master_key.backup

# Transferir para local seguro
scp /tmp/master_key.backup vault.empresa.com:/vault/

Ou via UI:

  1. Administração → Configuração.
  2. Clique Baixar master key.
  3. Arquivo master_key.txt é salvo no seu PC.
  4. Guarde em cofre seguro.

Recovery (disaster)​

Se você perdeu a key e não consegue mais acessar:

# Se tem backup:
docker compose cp /vault/master_key.backup centralops:/app/data/app_master_key
docker compose restart

# Se não tem backup:
# Você perdeu acesso às credenciais cifradas
# Opção: re-criar todas as integrações com novos secrets (sem downtime)

Auditoria de secrets​

Admin pode ver quais integrations têm secrets:

Vá a Administração → Configuração → Ver secrets.

Tabela:
Integration        | Platform | Has Secret | Last Updated
Sophos - Acme      | sophos   | ✓         | 2026-04-25 10:30
Defender - Acme    | defender | ✓         | 2026-04-20 14:15
Wazuh - Internal   | wazuh    | ✓         | 2026-04-22 09:00

Nota: você não vê o valor do secret (apenas que existe). Para segurança, valores nunca aparecem em UI.

Reset de secret (renovação de credencial)​

Se uma credencial foi comprometida (ex: Client Secret vaza):

  1. Gere nova credencial no vendor (ex: novo Client Secret em Sophos).
  2. Vá a Administração → Integrações → clique integração.
  3. Clique Editar.
  4. Preencha novo secret.
  5. Clique Salvar.

Sistema:

  • Decifra secret antigo.
  • Cifra novo secret.
  • Valida novo secret com vendor (test connection).
  • Se OK, substitui. Se erro, mantém antigo.

Senha de usuário​

Nota: senhas de usuário não usam APP_MASTER_KEY. Usam bcrypt (one-way hash).

  • Não podem ser recuperadas (nem admin).
  • Podem ser resetadas (via link enviado por e-mail ou manualmente).

Conformidade​

LGPD/GDPR​

  • Secrets são criptografados em repouso.
  • Key rotation deixa trilha em audit log.
  • Encryption key é derivada (não armazenada in plain text).
  • Backup da key antiga preservado por 90 dias (recovery).

PCI DSS​

  • AES-256 cumpre padrão.
  • GCM mode fornece integridade + confidencialidade.
  • Key derivation usa salt único por operação.

Troubleshooting​

"APP_MASTER_KEY inválida"​

Erro: "Invalid master key format" ou "Decryption failed"
Causa: key foi alterada ou corrompida
Ação: verifique .env, confirme 64 caracteres hex

"Depois de rotar key, integrations não funcionam"​

Erro: "Failed to decrypt secret" em logs
Causa: rotação falhou ou foi interrompida
Ação: 
  1. Verifique logs: docker compose logs centralops | grep decrypt
  2. Se parcial: retry rotação
  3. Se total: restore backup da key antiga, retry

"Perdi a master key, preciso acessar secrets"​

Opção 1: Se tem backup:
  - Restore backup em /app/data/app_master_key
  - Restart
  
Opção 2: Se não tem backup:
  - Gere nova key
  - Re-crie integrações (sem perda de dados históricos, só credenciais)
  - Avise aos clientes para re-autorizar se necessário

Boas práticas​

  1. Guarde APP_MASTER_KEY separado do .env (Vault, KMS, etc).
  2. Rotate annually — mínimo 1 vez por ano.
  3. Backup after rotation — armazene offline.
  4. Audit logs — revise regularmente quem rotated key.
  5. Credenciais dos vendors — nunca compartilhe via Slack/email (use secrets UI só).
