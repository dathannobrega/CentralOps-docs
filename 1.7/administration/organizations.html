Versão: 1.7 (legada)

Organizações

Isolamento de dados para MSSP (Managed Security Service Provider) com múltiplos clientes.

Público: Admin only.

Conceito​

Uma Organization agrupa:

  • Integrations (credenciais de vendor de 1 cliente).
  • Mappings (regras de normalização personalizadas).
  • Users (time local do cliente).
  • Audit logs (rastreamento de ações).
  • Retention policies (política de dados própria).

Dados de Org A são completamente invisíveis para Org B, mesmo na mesma instância.

Caso de uso: MSSP com 50 clientes​

Antes (sem multi-tenant):

  • 50 instâncias CentralOps (caro, operacional).
  • Ou 1 instância sem isolamento (compliance violation).

Depois (com multi-tenant):

  • 1 instância.
  • 50 orgs.
  • Admin gerencia globalmente.
  • Cada cliente vê só seus dados.

Acessar administração de orgs​

Vá a Administração → Organizações (menu de admin).

Interface​

Lista de orgs​

Tabela com:

ColunaDescrição
NomeNome da org (ex: "Acme Corp").
SlugIdentificador único (ex: "acme-corp").
UsuáriosQuantos users pertencem.
IntegraçõesQuantas integrations ativas.
CriadaTimestamp de quando foi criada.
AçõesEditar, deletar.

Criar nova organização​

Passo 1: Clique "+ Nova org"​

Passo 2: Preencha formulário​

Nome:        Acme Corporation
Slug:        acme-corp (gerado automaticamente de Name, editável)
Descrição:   Opção (ex: "Cliente financeiro, LGPD-critical")

Slug:

  • Único globalmente.
  • Lowercase, hífens (sem spaces).
  • Usado em URLs e API routes.

Passo 3: Criar​

Clique Criar org.

Transferir usuário para org​

Cenário: novo cliente necessita conta​

  1. Crie usuário normalmente (Usuários).
  2. Escolha org: "acme-corp".
  3. Salve.

Usuário agora só vê dados de "acme-corp".

Mover usuário entre orgs (Admin)​

Nota: não há UI para "mover". Criar nova conta em org destino é recomendado.

  1. Admin desativa conta na org origem.
  2. Admin cria nova conta na org destino.
  3. Compartilha link reset com usuário.

Configurar retenção por org​

Cada org pode ter política de retenção independente.

Acessar retenção​

  1. Clique org na lista → expande detalhe.
  2. Clique Editar retenção.

Campos​

Quarentena:      7 dias (padrão)
Drift:          90 dias
Histórico:      30 dias
Search results:  7 dias
Audit log:     365 dias (mínimo legal)

Validação:

  • Mínimo: 1 dia.
  • Máximo: 3650 dias (10 anos).
  • Audit log: nunca menos que 365 dias (compliance).

Efeito:

  • Eventos mais antigos são deletados automaticamente.
  • Job executa 3am UTC diariamente.

Exemplo: se quarentena = 14 dias, eventos > 14 dias são deletados.

Data deletion (LGPD/GDPR)​

Solicitação de purge​

Admin recebe solicitação: "Cliente X cancelou contrato. Delete todos os dados."

  1. Clique org → Solicitar purge de dados.
  2. Confirmação (segurança): "Tem certeza?"
  3. Digite DELETAR acme-corp (exato).
  4. Motivo: "Cliente cancelado 2026-04-25".
  5. Clique Confirmar.

Status do job​

Sistema cria job de deleção assíncrono:

Status: pending → running → completed

Você pode monitorar em Auditoria (procure por data_deletion).

O que é deletado​

  • Integrations (credenciais).
  • Mappings.
  • Events (raw, normalized, quarantined).
  • Users.
  • Organization itself.

O que NÃO é deletado​

  • Audit log master: arquivo JSON em /var/log/centralops/deletions/{job_id}.json. Sobrevive para trilha legal (LGPD art. 15).
  • Wazuh Indexer: eventos já enviados. Delete via Wazuh diretamente.

Limitações​

  • Sem hierarquia: hoje, orgs são flat. Parent/child é roadmap.
  • Sem grupo de orgs: não há forma de "admin de região" gerenciar subset de orgs.
  • Sem quotas: sem limites de usuários/events por org (roadmap).

Segurança​

  • Isolamento de query: SQL queries automaticamente filtram por organization_id.
  • RBAC respeitado: usuário de org A não pode atribuir permissão em org B (risco bloqueado).
  • Audit: cada mudança de org é registrada com quem, quando.

Casos de uso​

MSSP 50 clientes​

Cada cliente tem:

  • 1 org.
  • N users (local team).
  • N integrations (credenciais).
  • Custom mappings.
  • Own retention policy.

Admin central gerencia tudo de 1 dashboard.

Departamentos internos (empresa única)​

Mesmo conceito:

  • Org "Vendas".
  • Org "Financeiro".
  • Org "RH".

Cada um gerencia suas integrações, sem cruzamento de dados.

Vínculo com DFIR-IRIS​

Cada Organization precisa de um customer no DFIR-IRIS (iris_customer_id) para entregar eventos — sem ele, os eventos vão para a quarentena (missing_customer_id). O vínculo é automático para tenants do Sophos Partner e manual (via API) para as demais. Procedimento completo em DFIR-IRIS.

Próximos passos​

  • Usuários dessa org? Vá a Usuários.
  • Integrações dessa org? Vá a Administração → Integrações.
  • Vincular ao IRIS? Vá a DFIR-IRIS.
  • Retenção de dados? Vá a Compliance > Retenção.
