Retenção de dados

Configuração de quanto tempo dados ficam armazenados antes de serem deletados automaticamente.

Público: Admin (configurar), Compliance officer (auditoria).

Política padrão​

Global defaults (novas organizações):

Tipo de dadoRetençãoRazão
Quarentena7 diasEventos com erro, não importam muito histórico
Drift90 diasDescoberta de novos campos, menos urgente
Histórico (audit)30 diasLogs de chamadas da API
Search results7 diasCache de buscas, pode ser recalculado
Audit logs365 dias (hardcoded)LGPD/GDPR: mínimo 1 ano, imutável

Configurar por organização​

Cada organização pode sobrescrever defaults.

Acessar configuração​

Administração → Organizações → clique org → Editar retenção.

Formulário​

Quarentena (dias):       7       (min: 1, max: 3650)
Drift (dias):           90       (min: 1, max: 3650)
Histórico (dias):       30       (min: 1, max: 3650)
Search results (dias):   7       (min: 1, max: 3650)
Audit log (dias):      365       (não editável, obrigatório 1 ano)

Validação:

  • Mínimo 1 dia.
  • Máximo 3650 dias (10 anos).
  • Audit log: nunca menos que 365 dias (compliance).

Salvar​

Clique Salvar.

Aplicado na próxima execução de limpeza (3am UTC).

Como funciona limpeza​

Job agendado​

Celery Beat executa job "prune_all" diariamente às 3am UTC:

crontab(hour=3, minute=0)

Tasks individuais​

Cada tipo tem sua task:

TaskTabelaCritérioConfig
prune_expired_quarantinequarantine_eventscreated_at < now - X dayspor org
prune_expired_driftunknown_fieldslast_seen < now - X dayspor org + vendor
prune_expired_historyhistorytimestamp < now - X dayspor org
prune_expired_search_resultssearch_resultscreated_at < now - X dayspor org
prune_expired_audit_logsaudit_logscreated_at < now - X daysglobal (365 min)

Exemplo​

Org "Acme" com quarantine_retention = 14 dias:

1. Job inicia 3am UTC
2. Lê config da org: quarantine_retention = 14 dias
3. Query: SELECT * FROM quarantine_events 
         WHERE organization_id = acme_id 
         AND created_at < now() - interval '14 days'
4. DELETE esses rows
5. Log: "Deleted 1250 quarantine events older than 14 days"
6. Continua com drift, history, etc
7. Job concluído ~1 min (bulk delete é rápido)

Verificar execução​

Via logs​

docker compose logs centralops | grep -i "prune\|retention" | tail -20

Exemplo log:

2026-04-27 03:00:05 [INFO] Prune job started
2026-04-27 03:00:10 [INFO] Pruned 1250 quarantine_events
2026-04-27 03:00:15 [INFO] Pruned 50 unknown_fields
2026-04-27 03:00:20 [INFO] Pruned 300 history entries
2026-04-27 03:00:25 [INFO] Prune job completed

Via Flower (Celery UI)​

http://localhost:5555 → Tasks → search "prune_all"

Mostra execução e duração.

Via SQL​

-- Últimas execuções de limpeza
SELECT * FROM audit_logs 
WHERE action = 'prune'
ORDER BY created_at DESC
LIMIT 10;

-- Verificar tamanho de tabelas
SELECT 
  schemaname,
  tablename,
  pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename)) AS size
FROM pg_tables
WHERE tablename IN ('quarantine_events', 'unknown_fields', 'history')
ORDER BY size DESC;

Executar limpeza manualmente​

Se você quer rodar fora do horário 3am:

Via Celery CLI​

docker compose exec collector-worker-priority \
  celery -A backend.app.collectors.celery_app \
  call backend.app.collectors.retention_tasks.prune_all

Executa imediatamente, retorna status.

Auditoria de retenção​

Quem mudou retenção?​

SELECT 
  username,
  action,
  detail,
  created_at
FROM audit_logs
WHERE action = 'update_retention_config'
ORDER BY created_at DESC;

Registra: usuário, organização, valores novos.

Verificar se limpeza está rodando​

# Verificar se arquivo de lock do job existe
docker compose exec redis redis-cli GET "lock:prune_all"
# Se existe (valor = 1): job está rodando agora
# Se não existe: job não está rodando

Retenção vs. compliance​

LGPD/GDPR​

  • Direito ao esquecimento: após data deletion request, dados deletados < 24h.
  • Audit trail: audit logs não podem ser deletados antes de 1 ano (imutável).
  • Backup: responsabilidade de operações (fora scope CentralOps).

PCI DSS​

  • Cardholder data: se houver (raro), retenção 1 ano mínimo.
  • Log retenção: 1 ano, conforme padrão.

Performance​

  • Delete speed: ~10k rows/s (depend de DB).
  • Duration: ~1 min para org com 1M+ eventos (paralelo).
  • Downtime: 0 (não bloqueia queries de usuários).

Troubleshooting​

"Limpeza não rodou"​

# Verificar se job está no schedule
docker compose logs collector-beat | grep prune_all

# Se não vê: schedule pode estar corrompido
# Solução: docker compose restart collector-beat

"Dados antigos ainda presentes"​

# Verificar config da org
SELECT quarantine_retention_days 
FROM retention_configs 
WHERE organization_id = X;

# Se mostra 365 dias: vai levar 365 dias para deletar
# Alterar manualmente se necessário:
UPDATE retention_configs 
SET quarantine_retention_days = 30 
WHERE organization_id = X;

"Limpeza muito lenta"​

# Aumentar parallelismo do job
# (roadmap: configuração de N workers em paralelo)

# Ou manualmente: DELETE em pequenos batches
DELETE FROM quarantine_events 
WHERE organization_id = X 
AND created_at < now() - interval '7 days'
LIMIT 10000;
