LGPD / GDPR — Direito ao Esquecimento

Procedimento para deletar todos os dados de uma organização (cliente) conforme LGPD/GDPR art. 17.

Público: Admin only.

Conceito​

LGPD (Brasil) e GDPR (EU) exigem que você delete todos os dados de um indivíduo/entidade quando solicitado.

CentralOps implementa "direito ao esquecimento" em nível de organização:

Solicitação: "Acme Corp cancelou contrato, delete tudo"
→ DELETE todas as organizações/integrações/eventos de Acme
→ Resultado: zero traço de Acme no banco

Pré-requisitos para data deletion​

  1. Confirmação escrita: contato do cliente solicitando deleção.
  2. Motivo documentado: "contrato cancelado", "GDPR deletion request", etc.
  3. Backup offsite: você já fez backup (fora escopo CentralOps)?

Procedure​

Passo 1: Acessar administração​

Vá a Administração → Organizações.

Localize organização a deletar (ex: "Acme Corp").

Passo 2: Solicitar data deletion​

Clique organização → clique "Solicitar purge de dados" (botão vermelho).

Passo 3: Confirmação de segurança​

Modal aparece:

Aviso: Esta ação é irreversível!
Organização: Acme Corp
Slug: acme-corp
Dados a deletar: todos os eventos, integrations, users
Backup feito? (checkbox)

Checklist:

  • Você tem backup offsite?
  • Você notificou o cliente de que será deletado?

Se ambos OK, proceda.

Passo 4: Digite confirmação​

Campo de texto:

Digite EXATAMENTE: DELETAR acme-corp

(se digitar errado, cancela por segurança)

Digite com cuidado. Sistema verifica character-by-character.

Passo 5: Preenchimento de motivo​

Motivo da deleção:
[textarea] Cliente cancelou contrato 2026-04-27, LGPD request

Obrigatório (auditoria).

Passo 6: Enviar​

Clique "Confirmar purge".

Execução​

Sistema cria job de deleção assíncrono:

Status: pending → running → completed
Job ID: uuid-xyz-123
Celery Task ID: abcdef123456

O que é deletado​

TabelaCritério
integrationsorganization_id = X
mapping_definitionsorganization_id = X
mapping_versions(via cascade)
raw_eventsorganization_id = X
normalized_eventsorganization_id = X
quarantine_eventsorganization_id = X
unknown_fields (drift)(best-effort, particionado por vendor)
history (audit)organization_id = X
app_usersorganization_id = X
organizationsid = X
Rediscursores, token cache, dedupe, schedule (by integration_id)

O que NÃO é deletado​

  • Audit log master: arquivo JSON em /var/log/centralops/data-deletion/{job_id}.json

    • Sobrevive à deleção do DB.
    • Prova legal que deleção foi executada.
    • Acessível apenas a admin (offline).

  • Wazuh Indexer: eventos já enviados para Wazuh.

    • CentralOps não tem acesso para deletar.
    • Você deve deletar via Kibana: DELETE wazuh-alerts-* com critério customer_id = X.

  • Backups offsite: responsabilidade de operações.

Monitoramento do job​

Status em tempo real​

Vá a Administração → Organizações → Data Deletion Jobs (roadmap):

job-uuid-123     | acme-corp | running   | 10:05:00   | -
job-uuid-124     | old-corp  | completed | 09:30:00   | 09:32:15

Clique job → vê detalhes:

Status: running

  - [ ] Delete raw_events (done)
  - [ ] Delete normalized_events (in progress, 45%)

SQL query​

-- Ver jobs de deleção
SELECT 
  id,
  organization_slug,
  status,
  requested_at,
  completed_at,
  last_error
FROM data_deletion_jobs
ORDER BY requested_at DESC;

-- Ver qual org foi deletada
SELECT COUNT(*) FROM organizations WHERE slug = 'acme-corp';
-- Deve retornar 0

Verificação pós-deleção​

Confirmar que tudo foi deletado:

# 1. No BD
psql -U user -d centralops << EOF
SELECT COUNT(*) FROM organizations WHERE slug = 'acme-corp';
-- 0 rows ✓

SELECT COUNT(*) FROM app_users WHERE organization_id = 42;
-- 0 rows ✓

SELECT COUNT(*) FROM integrations WHERE organization_id = 42;
-- 0 rows ✓
EOF

# 2. Em Redis
redis-cli KEYS "collection:cursor:*:acme*"
# Deve estar vazio ✓

# 3. Audit log master (deve existir)
ls -la /var/log/centralops/data-deletion/
# Deve ter arquivo {job_id}.json ✓

SLA de deleção​

  • RTO: 24h (job inicia assim que worker disponível, nunca > 1h).
  • RPO: dados deletados são irrecuperáveis (sem backup offsite).

Auditoria pós-deleção​

Audit log master (/var/log/centralops/data-deletion/{job_id}.json) contém:

{
  "job_id": "uuid-xyz",
  "organization_id": 42,
  "organization_slug": "acme-corp",
  "requested_by": "admin-user",
  "requested_at": "2026-04-27T10:05:00Z",
  "reason": "Cliente cancelou contrato",
  "status": "completed",
  "completed_at": "2026-04-27T10:05:52Z",

    "users": 3,
    "organizations": 1

Salvar este arquivo:

  • Para GRC/compliance.
  • Para prova de execução.
  • Offline (arquivos, USB, etc).

Casos de uso​

Caso 1: Cliente cancela contrato​

Email: "Acme Corp deixa de ser cliente, delete dados"
Ação:
  1. Documentar solicitação (forward email)
  2. Ir a Admin → Orgs → Acme
  3. Solicitar purge
  4. Confirmar
  5. Aguardar < 1h (job completa)
  6. Download audit log master
  7. Email para cliente: "Seus dados foram deletados em [data], proof attached"

Caso 2: GDPR deletion request​

Email: "Data subject X requests deletion per GDPR art. 17"
Ação:
  1. Confirmar que indivíduo é "responsável de processamento" (parte de org)
  2. Se SIM: deletar toda org (para simplificar)
  3. Se NÃO (apenas 1 pessoa em org com N pessoas):
     - Separar pessoa em nova org (roadmap)
     - Deletar nova org

Caso 3: Backup disaster recovery​

Você deletou org acidentalmente
Recuperação:
  1. Restaurar DB do backup (< 1h antes)
  2. Org reaparece
  3. Log: "Org restaurada via disaster recovery"

Troubleshooting​

"Job ficou em 'running' por > 2h"​

Causa: worker crash, deadlock no DB, ou muitos dados
Ação:
  1. Verificar logs: docker compose logs centralops | grep deletion
  2. Se crash: reiniciar worker
  3. Se deadlock: kill job, retry (roadmap)

"Org não foi totalmente deletada"​

Sintoma: ainda existem registros com organization_id = X
Ação:
  1. Job retornou status "partial"?
  2. Vê error message?
  3. Possível cause: Wazuh Indexer indisponível (Wazuh purge skipped)
  4. Solução: deletar manualmente no Wazuh, retry CentralOps job

Compliance checklist​

  • Data deletion request documentada (email, form, etc).
  • Backup offsite feito antes.
  • Job completado com sucesso.
  • Audit log master salvo offline.
  • Wazuh Indexer also purged (if applicable).
  • Cliente notificado de conclusão.
  • GRC informado (log sent to compliance office).
