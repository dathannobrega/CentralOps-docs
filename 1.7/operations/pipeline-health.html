Saúde do Pipeline

Monitoramento em tempo real de integrações, workers Celery e envio para Wazuh.

Público: Viewer+. Admin vê métricas avançadas.

Visão geral​

Painel com 3 seções principais:

  1. Integrações — status de coleta por vendor.
  2. Collectors — saúde de workers Celery.
  3. Dispatcher — envio para Wazuh.

Integrações​

Métricas por integração​

Tabela com:

ColunaDescrição
NomeNome da integração (ex: "Sophos - Acme").
PlataformaVendor (sophos, defender, ninjaone).
Estadoactive, paused, error, awaiting_first_collection.
Última coletaTimestamp de quando último evento foi coletado.
Taxa (eventos/min)Velocidade média últimas 5 min.
SaúdeVerde (OK), Amarelo (warning), Vermelho (error).

Status de saúde​

  • Verde: coleta rodando, última coleta < 5 min atrás, sem erros.
  • Amarelo: último evento > 5 min, ou taxa de erro < 5%, ou taxa limite Sophos/Defender (429).
  • Vermelho: integração parada > 10 min, ou taxa de erro > 5%, ou credenciais inválidas.

Ações rápidas​

Clique em uma integração:

Painel lateral abre com:
- Detalhes (nome, platform, org)
- Último erro (se houver)
- Histórico de coleta (gráfico 24h)
- Botão: Pausar / Retomar
- Botão: Testar conexão novamente
- Botão: Ver logs recentes

Ação comum: "Integração parada"

  1. Vê status Vermelho.
  2. Clica integração.
  3. Lê último erro (ex: "401 Unauthorized").
  4. Clica "Testar conexão".
  5. Se falha: credenciais expiradas. Vá a Administração → Integrações para rotacionar.
  6. Se passa: erro transiente. Clica "Retomar".

Collectors​

Workers Celery​

Tabela mostrando:

ColunaDescrição
WorkerNome (collector-worker-priority-1, etc).
Statusonline, offline.
Tasks ativasN tasks em execução.
Taxa (tasks/min)Velocidade média.
MemóriaRAM usada.
UptimeQuanto tempo está rodando.

Filas​

Dropdowns mostrando depth de cada fila:

  • collect.priority — coletas urgentes (Sophos alerts, Defender).
  • collect.bulk — coletas batch (NinjaOne, XDR queries).
  • dispatch — envio para Wazuh.
  • maintenance — limpeza de retenção.

Interpretação:

  • Depth 0 = fila vazia (todos os tasks acabaram).
  • Depth > 100 = fila crescendo (workers mais lentos que produtor). Escalação recomendada.
  • Depth estável alto = load balanceado (esperado em alta throughput).

Gráficos​

Histórico das últimas 24h:

  • Tasks/min: velocidade de processamento.
  • Queue depth: tamanho das filas.
  • Errors/min: quantos tasks falharam por minuto.

Leitura:

Se gráfico mostra "errors/min" > 0:
- Integrações com problema?
- Vendor retorna taxa limite (429)?
- Mappings falhando?

Dispatcher​

Envio para Wazuh:

MétricaDescrição
StatusConectado, desconectado, erro de TLS.
Taxa (eventos/min)Velocidade de despacho.
LatênciaTempo médio para enviar 1 evento.
Falhas (últimas 24h)Quantos eventos falharam envio.
Último eventoTimestamp de quando foi despachado.

Verde: conectado, taxa normal, sem falhas.

Amarelo: latência alta (>1s por evento) ou falhas < 1%.

Vermelho: desconectado de Wazuh, ou falhas > 1%.

Ação: "Dispatcher offline"

  1. Vê status Vermelho, "desconectado de Wazuh".
  2. Clique painel → vê erro (ex: "Connection refused").
  3. Checklist:
    • Wazuh Manager está rodando? curl -k https://wazuh:6514
    • Rede OK? ping wazuh
    • Certificado TLS válido? (se usar CA custom)
    • Firewall bloqueando?
  4. Se Wazuh OK, reinicie dispatcher: docker compose restart collector-dispatcher.

Casos de uso​

Monitoramento matinal​

  1. Abre Pipeline Health.
  2. "Todas as integrações em verde?" — sim = segue o dia.
  3. "Collector workers online?" — sim.
  4. "Dispatcher conectado?" — sim.
  5. Se qualquer coisa amarela/vermelha, investiga antes de escalar.

Diagnóstico: "Não estou vendo eventos"​

  1. Vê integração em estado "active".
  2. Última coleta: 5 min atrás (OK).
  3. Mas nenhum evento em Alertas.
  4. Possibilidades:
    • Evento em quarentena? Vá a Quarentena.
    • Taxa é 0? Integração não está coletando nada (vendor tem eventos?).
    • Dispatcher offline? Evento foi coletado mas não chegou ao Wazuh.

Escalação: "Integração sempre vermelha"​

  1. Clique integração.
  2. Lê error: "429 Too Many Requests".
  3. Significa: taxa limite do vendor atingida.
  4. Ações:
    • Reduzir frequência de coleta (padrão 1-5 min, pode aumentar).
    • Contatar vendor sobre limite.
    • Escalar para Engineer/Admin.

Limitações​

  • Delay: métricas atualizam a cada 1-2 min. Não é real-time.
  • Histórico: apenas últimas 24h. Para tendências, use Prometheus/Grafana (roadmap).
  • Acesso: Viewer vê tudo, mas Admin vê logs e erros detalhados.

Próximos passos​

  • Integração com problema? Vá a Administração → Integrações.
  • Taxa limite? Vá a runbook Dispatcher.
  • SLO em risco? Vá a runbook SLO Burn.
