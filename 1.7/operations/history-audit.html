Histórico e Auditoria

Rastreabilidade completa de quem fez o quê e quando. Imutável e retenção 365 dias (LGPD/GDPR).

Público: Viewer+. Admin vê logs de sistema também.

Tipos de eventos auditados​

Integrations​

  • Criação de integração.
  • Edição de credenciais.
  • Pausa/retomada.
  • Teste de conexão.
  • Deletção.

Mappings​

  • Criação de mapping.
  • Edição de DSL.
  • Rollback para versão anterior.
  • Publicação (marcar como current).

Quarantine​

  • Descarte de evento.
  • Reprocessamento de evento.
  • Bulk discard.
  • Bulk reprocess.

Drift​

  • Marca campo como ignorado.
  • Marca campo como mapeado.
  • Deletção de entrada drift.

Users (Admin only)​

  • Criação de usuário.
  • Alteração de papel.
  • Desativação.
  • Reset de senha.

Organizations (Admin only)​

  • Criação de org.
  • Alteração de configuração.
  • Retenção policy change.
  • Data deletion request.

System (Admin only)​

  • APP_MASTER_KEY rotation.
  • Backup/restore.
  • Config changes (SMTP, etc).

Interface​

Filtros​

FiltroOpções
Tipointegration, mapping, quarantine, user, etc.
Açãocreate, edit, delete, pause, test, etc.
UsuárioDropdown de todos os usuários ativos.
DataPicker: últimas 24h, 7 dias, 30 dias, custom.

Colunas​

ColunaConteúdo
TimestampQuando (ISO 8601, UTC).
UsuárioQuem fez (ou "System" se automático).
TipoResource type (integration, mapping, user).
AçãoO que foi feito (create, edit, delete, etc).
RecursoNome/ID do que foi alterado.
DetalhesMudanças específicas (expand).

Workflow​

1. Investigar mudança recente​

"Alguém alterou mapping de Sophos?"

Filtro: Type = "mapping", Action = "edit"
Resultado: lista de edições de mappings
Encontra: "Sophos alerts v3 → v4" editado às 10:30 por "alice"
Clique: expande detalhe
Vê: "Added rule: severity = if alert_level >= 5..."

2. Auditoria de usuário​

"Quais ações alice fez esta semana?"

Filtro: User = "alice", Data = "últimos 7 dias"
Resultado: todas as ações de alice
Revisa: 3 edições de mapping, 2 discards em quarentena, 1 pausa integração

3. Investigação de incidente​

"Quando foi criada integração 'Backdoor Defender'?"

Filtro: Type = "integration", Recurso = "Backdoor" (busca livre)
Resultado: criação da integração às 2026-04-25 14:32 por "bob"
Timestamp próximo a alerta no SIEM?
Possível backdoor criado durante ataque

4. Auditoria de compliance​

"Quem acessou dados de cliente X?"

Filtro: Organization = "cliente-x"
Resultado: todas as ações em cliente-x
Exporta: relatório para GRC

Download/exportação​

Botão Download exporta resultado em:

  • CSV: planilha (Excel, Sheets).
  • JSON: raw events.
  • PDF: relatório formatado (para impressão/compliance).

Campos inclusos: timestamp, user, type, action, resource, details.

Útil para:

  • Entrega a auditor externo.
  • Compliance report mensal.
  • Evidência em forensics.

Retenção e limpeza​

Policy​

  • Audit logs: retenção 365 dias (hardcoded, não configurável).
  • Retenção após 365 dias: deletado automaticamente (job noturno).
  • Exceção: eventos com user_id = NULL (sistema) sempre retidos por 7 anos (legislativo).

Verificar limpeza​

-- Últimos audit logs
SELECT * FROM audit_logs ORDER BY created_at DESC LIMIT 10;

-- Deletados em período
SELECT COUNT(*) FROM audit_logs WHERE created_at < now() - interval '365 days';

Restrições de acesso​

PapelPode ver
ViewerTudo (read-only) exceto logs de admin (user create, org changes).
OperatorTudo de Viewer + seus próprios discards/reprocess.
EngineerTudo de Operator + mapping edits.
AdminTudo (inclui user management, system logs).

Cross-organization: usuário vê só logs da sua org (exceto admin, que vê todas).

Casos de uso​

Forensics pós-incidente​

  1. Período: antes/depois do incidente.
  2. Filtro: tudo o que aconteceu.
  3. Analisa timeline:
    • Quando foi criada integração maliciosa?
    • Quem acessou dados sensíveis?
    • Qual mapping foi alterado para esconder eventos?

Auditoria anual​

  1. Download audit logs: ano anterior (ex: 2025).
  2. Filtra: operações críticas (user create, data deletion, key rotation).
  3. Entrega a GRC para attestation.

Change management​

  1. Mapeamento foi editado: quando, por quem, o quê mudou?
  2. Rollback foi feito: por quem, a qual versão?
  3. Comparação com changelog do vendor (alinhado?).

Limitações​

  • Delay: eventos aparecem em < 1 min.
  • Granularidade: uma ação = um log. Bulk operations geram N logs.
  • Dados pessoais: audit log pode conter (username, email). Se GDPR data deletion, logs não são deletados (compliance rule).

Próximos passos​

  • Precisa investigar mudança? Use filtros acima.
  • Compliance report? Clique Download → PDF.
  • Forensics profunda? Vá a Runbooks para investigação passo-a-passo.
