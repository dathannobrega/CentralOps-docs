Quarentena

Eventos que falharam parsing ou validação DSL aparecem aqui. Você triage: descarta (ignora) ou reprocessa (após fix).

Público: Operator+. Viewer vê mas não executa ações.

O que vai para quarentena​

Qualquer evento que:

  1. Falha parse (JSON inválido, encoding errado).
  2. Falha validação de mapping (campo obrigatório missing, type cast error).
  3. Falha schema (severity não é válido, timestamp inválido).
  4. Sem customer_id — a Organization não tem iris_customer_id vinculado (error_kind="missing_customer_id"). Resolve-se vinculando o customer no IRIS: ver DFIR-IRIS.

Não vai para quarentena: eventos que passam na validação vão direto para normalizado → dispatcher → Wazuh.

Workflow de triage​

Passo 1: Abrir quarentena​

Vá a Operação → Quarentena.

Vê lista com:

  • Timestamp (quando recebido).
  • Plataforma (Sophos, Defender, etc).
  • Erro (breve descrição).
  • Ação (expand, discard, reprocess).

Passo 2: Expandir e revisar detalhe​

  1. Clique em um evento.
  2. Veja raw JSON (payload original do vendor).
  3. Veja erro específico (ex: "Required field 'timestamp' missing" ou "Type error in field 'severity'").
  4. Analise: é bug do mapping ou anomalia do vendor?

Passo 3: Decidir ação​

Ação A: Discard​

"Este evento é falso positivo / ruído / vendor bugado. Ignore."

Clique: Discard
Status: muda de "pending" para "discarded"
Resultado: evento não é reprocessado, nunca vai pro Wazuh

Use quando:

  • Vendor retornou evento malformado (não é bug do CentralOps).
  • Campo obrigatório faltando porque vendor fez change breaking.
  • Evento é spam/teste do vendor.

Ação B: Reprocess​

"Eu ajustei o mapping. Recoloca na fila para transforação nova."

Pré-requisito: você (ou um Engineer) editou o mapping
Clique: Reprocess
Status: muda para "pending"
Resultado: evento entra novamente na fila, transforma com novo mapping

Workflow completo:

  1. Veja evento X em quarentena com erro "Required field 'severity'".
  2. Clique Expand → JSON mostra campo alert_level (tem valor 5).
  3. Problema: mapping antigo esperava severity, mas Sophos mandou alert_level.
  4. Vá a Mappings → edite mapping sophos.alert.
  5. Adicione linha: severity = map(alert_level, {1: "low", 5: "critical"}).
  6. Salve novo mapping (v2).
  7. Volte a Quarentena → clique no evento X.
  8. Clique Reprocess.
  9. Sistema enfileira evento X com novo mapping v2.
  10. Após segundos: evento X sai de quarentena, aparece em Alertas normalizados.

Passo 4: Validar se funcionou​

Após reprocess:

  • Se sucesso: evento desaparece de quarentena, aparece em Alertas.
  • Se falha novamente: volta para quarentena com novo erro. Volte ao passo 2.

Filtros​

Por plataforma​

Dropdown: Sophos, Defender, NinjaOne.

Útil se quer triage por vendor (ex: "Todos os erros Sophos desta manhã").

Por tipo de erro​

  • parsing_error — JSON inválido.
  • validation_error — campo missing ou inválido.
  • type_error — type cast falhou (ex: string esperava int).
  • missing_customer_id — Organization sem iris_customer_id (ver DFIR-IRIS).

Por data​

Picker: "últimas 24h" (padrão), "últimos 7 dias", custom range.

Nota: quarentena é retenção 7 dias. Após 7 dias, evento é automaticamente deletado.

Casos de uso comuns​

Caso 1: "Sophos mudou seu schema"​

Sintoma: muitos eventos Sophos em quarentena com erro "field X missing".

Diagnóstico:

  1. Abra 3 eventos diferentes.
  2. Compare raw JSONs.
  3. Veja novo campo que antes não existia ou field que foi deletado.

Ação:

  1. Vá a Mappings → sophos.alert.
  2. Revise regras DSL.
  3. Adicione/remova campos conforme novo schema Sophos.
  4. Teste com 1 evento: Reprocess.
  5. Se OK, faça bulk reprocess de todos eventos Sophos (botão em quarentena).

Caso 2: "Falsos positivos de validação"​

Sintoma: eventos aparecem em quarentena com erro "Type error", mas field tem valor legítimo.

Diagnóstico:

  1. Clique evento.
  2. Veja field → JSON mostra string, mas mapping esperava int.

Ação:

  1. Vá a Mappings.
  2. Adicione type cast: severity = int(raw_severity_field) ou severity = severity_field as int.
  3. Reprocess evento.

Caso 3: "Campo obrigatório faltando"​

Sintoma: erro "Required field 'timestamp' missing".

Ação:

  1. Se campo existe no raw JSON com outro nome, remapeie: timestamp = received_at.
  2. Se campo não existe e vendor não suporta, forneça default: timestamp = now() (timestamp coleta).
  3. Reprocess e valide.

Bulk actions​

Se 100+ eventos em quarentena do mesmo tipo:

  1. Filtro por plataforma (ex: Sophos).
  2. Clique Ações em massa → Discard All.
  3. Ou: Reprocess All (após fix do mapping).

Observações​

  • Quarentena não é rejeição permanente. Você pode reprocess depois.
  • Audit log registra cada discard/reprocess com quem e quando.
  • Retenção: após 7 dias, evento é deletado automaticamente (configurável).

Próximos passos​

  • Muita quarentena? Vá a Troubleshooting Mappings.
  • Precisa editar mapping? Vá a Mappings.
  • Alta taxa de quarentena? Vá a runbook High Quarantine Rate.
