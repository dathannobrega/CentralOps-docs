Arquitetura

CentralOps é um security data pipeline (Extract-Transform-Load) para eventos de segurança, projetado pra SOC multi-tenant e MSSPs. Esta página explica, em linguagem de usuário, o que cada parte do sistema faz, como os workers trabalham, como escalá-los quando o volume cresce e como validar se algum está com problema.

Visão geral​

O pipeline tem três grandes momentos:

  1. Coletar eventos das APIs dos vendors (Sophos, Defender, NinjaOne, Wazuh — e qualquer outro registrado via register_provider).
  2. Normalizar o evento bruto para o schema OCSF canônico usando regras CML (CentralOps Mapping Language) versionadas.
  3. Despachar o evento normalizado para o destino configurado (Wazuh Manager via Syslog TCP/TLS por padrão; arquitetura _Target Protocol permite Splunk HEC, Elastic, S3, Sentinel ou Kafka adicionais).

Tudo isso roda em containers separados, coordenados por um broker (Redis) e um banco relacional (PostgreSQL):

Componentes — o que cada parte faz​

centralops (Backend API + UI)​

Container principal que serve a API REST (FastAPI) e a interface web (React). É onde o usuário faz login, configura integrações, edita mappings e vê dashboards. Não coleta nem despacha eventos — só comanda. Replicável horizontalmente atrás de um load balancer.

Celery Beat (scheduler)​

Singleton que agenda as tarefas periódicas de coleta. Lê o schedule do Redis (via RedBeat) e dispara tasks a cada 1–5 minutos por integração. Se você cria uma integração nova na UI, o Beat passa a agendá-la automaticamente em segundos — sem restart.

Por que singleton? Dois Beats agendariam a mesma task duas vezes. O Docker Compose mantém apenas 1 réplica e reinicia se cair.

collector-worker-priority​

Workers que coletam eventos de tempo real — alertas Sophos, detecções Defender. Latência baixa importa. Por padrão sobem 2 réplicas com concurrency=4 cada (= 8 tasks paralelas no total).

Cada worker:

  • Puxa task da fila collect.priority.
  • Carrega credenciais cifradas do Postgres, faz refresh OAuth se preciso.
  • Chama API do vendor a partir do cursor (último evento visto), evitando reprocessar.
  • Normaliza o evento aplicando o mapping ativo da DSL v2.
  • Envia evento normalizado para dispatch.wazuh (ou quarantine se falhar).

collector-worker-bulk​

Workers para coletas menos sensíveis a latência — auditoria, atividades NinjaOne — e também para backfills (reprocessar até 90 dias de eventos quando uma regra de mapping muda). Por padrão sobe 1 réplica com concurrency=8.

Consome duas filas: collect.bulk e collect.backfill. Backfills são longos por design — não devem competir com collect.priority.

collector-dispatcher​

Workers responsáveis por enviar o evento normalizado para o Wazuh via Syslog TCP/TLS. Por padrão 2 réplicas com concurrency=8.

Consome dispatch.wazuh (fila normal) e dispatch.dlq (Dead Letter Queue para retries). Se o Wazuh estiver indisponível, o dispatcher acumula em dispatch.dlq e tenta de novo com backoff — eventos não são perdidos enquanto a DLQ aguenta.

Redis​

Multifunção:

  • Broker do Celery (db 1) — armazena tasks pendentes nas filas.
  • Result backend (db 2) — guarda status de tasks.
  • Cache de cursor (db 0) — collection:cursor:{integration_id}:{stream}.
  • Cache de tokens OAuth — refresh tokens com TTL.
  • Dedup de eventos — dedupe:{integration_id} previne duplicatas em coletas concorrentes.
  • Schedule do Beat (RedBeat).

PostgreSQL​

Estado durável: integrações, organizações, usuários, mappings, audit log, eventos em quarentena, drift entries. Default do Compose. Suporta também SQLite via DATABASE_URL para dev.

Flower​

UI web em :5555 para monitorar workers Celery — tasks ativas, throughput, históricos. Em produção exige FLOWER_BASIC_AUTH=user:senha no .env (o serviço falha pra subir sem isso).

Workers em detalhe​

WorkerFila(s)ConcurrencyRéplicas (default)Variável p/ escalar
collector-worker-prioritycollect.priority42COLLECTOR_PRIORITY_REPLICAS
collector-worker-bulkcollect.bulk, collect.backfill81(escalar via --scale)
collector-dispatcherdispatch.wazuh, dispatch.dlq82COLLECTOR_DISPATCHER_REPLICAS
collector-beat(não consome filas)—1 (singleton)— não escalar

Concurrency = quantas tasks aquele processo executa em paralelo (modelo prefork, sem GIL). Réplicas = quantos processos existem do mesmo worker.

Capacidade total de coleta priority = replicas × concurrency = 2 × 4 = 8 coletas simultâneas por padrão.

Como escalar workers​

Em runtime (sem editar arquivo)​

# 4 réplicas de priority (4 × 4 = 16 coletas paralelas)
docker compose -f compose/docker-compose.yml up -d \
  --scale collector-worker-priority=4

# 4 réplicas de dispatcher (4 × 8 = 32 envios paralelos pro Wazuh)
docker compose -f compose/docker-compose.yml up -d \
  --scale collector-dispatcher=4

# 2 réplicas de bulk (útil durante backfill grande)
docker compose -f compose/docker-compose.yml up -d \
  --scale collector-worker-bulk=2

Novos workers começam a consumir tasks em menos de 1 segundo. Workers que saem terminam graciosamente — tasks em execução completam, cursores são salvos, sem perda de dados.

Permanente (via .env)​

# .env do projeto
COLLECTOR_PRIORITY_REPLICAS=4
COLLECTOR_DISPATCHER_REPLICAS=4

Aplica no próximo docker compose up -d.

Aumentar concorrência interna (avançado)​

Se um worker tem CPU/memória sobrando mas a fila ainda cresce, aumente a --concurrency no docker-compose.yml. Cuidado: mais concorrência = mais conexões simultâneas ao Postgres/Redis e à API do vendor. Pode bater rate limit do vendor antes de ganhar throughput.

Regra de bolso: escale réplicas primeiro, concurrency depois.

Como validar se um worker está com problema​

Há quatro caminhos complementares — use o que for mais rápido pro tipo de sintoma.

1. Pipeline Health (UI) — primeiro lugar a olhar​

Menu lateral → Saúde do Pipeline. Mostra:

  • Tabela de integrações com status (verde/amarelo/vermelho), última coleta, taxa eventos/min.
  • Tabela de workers Celery: nome, status (online/offline), tasks ativas, taxa, memória, uptime.
  • Profundidade de cada fila (collect.priority, collect.bulk, dispatch.wazuh, etc.).

Se uma fila está com profundidade crescente e workers ociosos → problema na task (erro repetido). Se profundidade cresce e workers a 100% → falta capacidade, escalar.

Detalhes em Saúde do Pipeline.

2. Flower — visão técnica de Celery​

Acesse http://<host>:5555 (login via FLOWER_BASIC_AUTH). Mostra:

  • Tasks ativas por worker, com tempo de execução em tempo real.
  • Histórico das últimas N tasks (sucesso, falha, retry).
  • Broker — fila por fila, mensagens pendentes.
  • Pode revogar task travada manualmente (uso responsável — em produção, só admin).

Caso típico: "task X aparece em 'active' há 10 minutos no Flower" → provável travamento em I/O (API do vendor sem timeout). Reinicie o worker que segura essa task.

3. Logs do container​

# tempo real
docker compose logs -f collector-worker-priority
docker compose logs -f collector-dispatcher

# últimas 200 linhas só de erros
docker compose logs --tail 200 collector-worker-priority | grep -i error

O que procurar:

  • Token refreshed for integration_id=X — saudável, OAuth funcionando.
  • Duplicate event skipped — saudável, dedup atuando.
  • 401 Unauthorized — credencial expirada (vai pra status "error" da integração).
  • 429 Too Many Requests — rate limit do vendor; backoff automático.
  • Quarantine: missing_required_field — mapping precisa de revisão.

4. Comandos diretos do Celery (último recurso)​

# entrar no container
docker compose exec collector-worker-priority bash

# todos os workers respondem a ping?
celery -A app.collectors.celery_app inspect ping

# listar tasks ativas em todos os workers
celery -A app.collectors.celery_app inspect active

# estatísticas de carga
celery -A app.collectors.celery_app inspect stats

Worker que não responde ao ping está travado ou crashou. Reinicie só ele:

docker compose restart collector-worker-priority

Casos de uso — quando e como escalar​

Caso 1 — Volume de eventos triplicou (Sophos liberou alertas detalhados)​

Sintomas:

  • Profundidade de collect.priority subindo continuamente em Saúde do Pipeline.
  • Workers priority com CPU em ~95%, memória estável.
  • Tempo entre coleta e dispatch sobe de segundos para minutos.

Diagnóstico:

A capacidade replicas × concurrency (= 8 default) ficou abaixo da demanda. Não é problema de mapping nem de Wazuh — só de coletar.

Ação:

# dobrar capacidade priority
docker compose up -d --scale collector-worker-priority=4

Em 1–2 minutos a fila começa a drenar. Se voltar a crescer, dobre dispatcher também — eventos coletados sem dispatcher acumulam em dispatch.wazuh.

Caso 2 — Backfill de 90 dias (mapping foi corrigido, precisa reprocessar)​

Sintomas:

  • Você disparou um job de backfill grande na UI.
  • Fila collect.backfill cresce. Bulk worker (1 réplica, concurrency 8) demora dias.
  • collect.priority segue normal — não foi afetada, é a parte boa do design.

Ação:

# subir bulk pra 3 réplicas TEMPORARIAMENTE
docker compose up -d --scale collector-worker-bulk=3

Quando terminar:

docker compose up -d --scale collector-worker-bulk=1

Backfill consome muito Postgres (insere RawEvent em massa). Se observar o Postgres em ~80% CPU, mantenha 1–2 réplicas e aceite mais tempo.

Caso 3 — Mappings caros (muitas regras, parsing pesado)​

Sintomas:

  • CPU dos workers de coleta alta mesmo com volume estável.
  • Throughput por evento caiu (tasks/min em Pipeline Health diminui).
  • Você sabe que mexeu recentemente em mappings — adicionou regras com when complexo, JMESPath aninhado, ou muitos array_builder.

Diagnóstico:

A normalização ficou cara. Coleta puxa rápido, mas processar um evento leva mais tempo. Escalar workers ajuda na largura de banda, mas você está pagando custo desnecessário por evento.

Ação (na ordem):

  1. Revisar o mapping em /mappings. Procurar:
    • Predicados when com vários or/and aninhados — simplifique.
    • value_map enormes (>50 entradas) — considere quebrar em vários mappings.
    • JMESPath com filtros caros tipo data[?level >= \5`].name— pode ser pré-calculado empreprocess`.
  2. Drift Explorer ajuda a achar regras inúteis: campos com count = 0 são candidatos a remoção.
  3. Só depois — se ainda for caro mas necessário — escalar workers priority pra absorver o custo: 
    docker compose up -d --scale collector-worker-priority=4

Mappings caros aparecem mais quando o vendor manda eventos com muitos campos opcionais. Otimize a regra antes de jogar máquina.

Caso 4 — Wazuh lento ou indisponível​

Sintomas:

  • collect.priority está OK; quem cresce é dispatch.wazuh.
  • Logs do collector-dispatcher com Connection refused ou TLS handshake timeout.
  • DLQ (dispatch.dlq) começa a receber mensagens.

Diagnóstico:

O Wazuh Manager parou de aceitar — manutenção, sobrecarga, certificado expirado. Os eventos não são perdidos enquanto Redis tem espaço, mas a janela acaba.

Ação:

  1. Validar Wazuh: nc -zv $WAZUH_SYSLOG_HOST 6514 do host do CentralOps.
  2. Se Wazuh está fora — investigar do lado dele primeiro.
  3. Se Wazuh voltou mas DLQ acumulou: 
    docker compose up -d --scale collector-dispatcher=4
    DLQ é drenada com retry automático.
  4. Confira WAZUH_CA_BUNDLE se for erro de TLS — certificado pode ter rotacionado.

Runbook detalhado: Dispatcher.

Caso 5 — Drift explodindo (vendor mudou schema sem avisar)​

Sintomas:

  • Página /drift cheia de campos com status new.
  • Cresce a cada coleta nova.
  • Eventos continuam chegando no Wazuh, mas com menos contexto do que tinha antes.

Diagnóstico:

O vendor adicionou ou renomeou campos. As regras atuais ignoram silenciosamente esses campos novos. Não é perda de evento, é perda de fidelidade — informação útil cai no chão.

Ação:

  1. Triagem em massa no Drift Explorer: filtre por vendor e event_type, selecione os campos que não te interessam (ex: campos de telemetria interna do vendor) e use bulk Ignorar.
  2. Os que interessam — abra o mapping correspondente em /mappings, adicione regras novas pra cobrir esses paths.
  3. Reprocesse eventos antigos via backfill se quiser preencher histórico.

Após Sophos/Microsoft fazerem updates de feature, drift explodir é esperado. O fluxo de filtrar + bulk-ignore + adicionar regra foi desenhado pra isso.

Garantias do pipeline​

  • Idempotência — cada evento tem message_id do vendor; Redis Set evita duplicação 24h. Coletar duas vezes é seguro.
  • Ordenação por integração — cursor (timestamp, id) mantém a ordem temporal dentro de uma integração; não há ordenação global entre vendors diferentes.
  • At-least-once delivery — falha de rede pode causar reentrega. Wazuh dedupa via event_id no envelope.
  • Sem perda silenciosa — falhas vão pra quarentena (validação) ou drift (campos não-mapeados). Você sempre vê.
