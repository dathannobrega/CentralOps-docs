Scheduler (RedBeat)

RedBeat permite agendar coletas de forma dinâmica (sem restart) via Redis.

Público: Engineer+ (entender). Admin (operar).

Como funciona​

Fluxo​

1. Admin cria integração em CentralOps
   └─ POST /api/integrations (platform=sophos, etc)

2. FastAPI backend registra integração em BD
   └─ Cria entry em Redis via RedBeat

3. Celery Beat lê schedule do Redis a cada 5s
   └─ Vê integração "sophos-acme", próxima execução 10:05 UTC

4. Beat dispara task "collect_vendor_logs_priority"
   └─ Args: [integration_id=42, stream="alerts"]

5. Worker Celery executa task
   └─ Collector puxa eventos via Sophos API
   └─ Armazena em PostgreSQL + Redis (cursor)

6. Próxima execução agendada
   └─ RedBeat calcula "now + 1 min"
   └─ Task entra em fila novamente

Sem restart​

Adicionar integração não requer restart de Beat ou workers.

Antes: integração criada → teria que reiniciar Beat para aparecer no schedule
Agora: integração criada → <5s depois aparece no schedule do Beat

RedBeat no Redis​

Dados persistidos em Redis:

redis-cli
> KEYS "redbeat::schedule:*"
1) "redbeat::schedule:sophos-acme-alerts"
2) "redbeat::schedule:sophos-acme-cases"
3) "redbeat::schedule:defender-acme-incidents"
...

> HGETALL "redbeat::schedule:sophos-acme-alerts"
1) "next_run_time"
2) "2026-04-27T10:05:30+00:00"
3) "task"
4) "collectors.collect_vendor_logs_priority"
5) "kwargs"
6) "{"integration_id": 42}"
...

Cada schedule entry tem:

  • task: nome da task Celery.
  • kwargs: argumentos (integration_id, stream).
  • next_run_time: próxima execução (ISO 8601).

Gerenciar schedules​

Ver schedules via CLI​

# Ver todas as entries
docker compose exec redis redis-cli KEYS "redbeat::schedule:*" | wc -l

# Inspeccionar uma entry
docker compose exec redis redis-cli HGETALL "redbeat::schedule:sophos-acme-alerts"

# Ver próxima execução
docker compose exec redis redis-cli HGET "redbeat::schedule:sophos-acme-alerts" "next_run_time"

Resetar schedule (forçar re-agendamento)​

Se integração está presa em "Aguardando primeira coleta":

# Deletar entry (força Beat a re-criar)
docker compose exec redis redis-cli DEL "redbeat::schedule:sophos-acme-alerts"

# Beat detecta falta em próximo ciclo (< 5s)
# Re-cria entry automaticamente

Pausar coleta​

Via UI: Administração → Integrações → clique integração → Pausar.

Backend deleta schedule entry. Volta ao clicar Retomar.

Frequências por vendor​

PlataformaStreamIntervalo

Configurável via backend/app/collectors/registry.py → schedule=timedelta(minutes=X).

Monitoramento​

Via Flower (Celery UI)​

# Acessar Flower
curl http://localhost:5555

# Tasks → Scheduled
# Ver todas as tasks agendadas

Via Celery CLI​

# Ver tasks agendadas
docker compose exec collector-beat \
  celery -A backend.app.collectors.celery_app inspect scheduled

# Saída: lista de tasks com ETA (estimated time of execution)

Via logs​

docker compose logs collector-beat | grep -i scheduled

Exemplo log:

2026-04-27 10:00:05 [INFO] Schedule changed by redbeat
2026-04-27 10:00:05 [INFO] Scheduler: Sending due task sophos-acme-alerts
2026-04-27 10:00:05 [INFO] Scheduler: Schedule: 10:01:05 UTC

Troubleshooting​

"Integração não coleta"​

Checklist:

  1. Entry no Redis?

    redis-cli KEYS "redbeat::schedule:*" | grep seu-vendor

  2. Beat está rodando?

    docker compose ps collector-beat

  3. Workers disponíveis?

    docker compose logs collector-worker-priority | tail -5

  4. Fila não entupida?

    redis-cli LLEN collect.priority
    # Se > 1000, workers lentos

Ação:

  • Se Beat parou: docker compose restart collector-beat
  • Se workers parou: docker compose restart collector-worker-priority
  • Se fila entupida: escale workers

"Schedule desapareceu do Redis"​

Causa: Redis foi reiniciado ou FLUSHDB.

Solução: reiniciar Beat (detecta missing entries, re-cria automaticamente).

docker compose restart collector-beat

"Próxima execução está no passado"​

Significa: task deveria ter rodado mas não rodou (worker lento ou indisponível).

Beat dispara imediatamente na próxima verificação (< 5s).

Boas práticas​

  1. Não deletar schedules manualmente — use UI (Pausar/Retomar).
  2. Não alterar intervalo em runtime — requer redeploy e reset de cursor.
  3. Monitorar Redis memory — muitas integrações = mais entries = mais RAM Redis.
  4. Backup Redis — schedules são críticos. Ativar RDB/AOF.

Performance​

  • Latência: schedule → task dispatch: < 1s.
  • Overhead: cada integração = 1 entry Redis (~100 bytes).
  • Limite prático: 1000+ integrações em 1 instância (testado).
