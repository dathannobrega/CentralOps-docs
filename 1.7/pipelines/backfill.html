Backfill (Recoleta retroativa)

Recolher eventos históricos de integrações para preenchimento de gaps ou teste.

Público: Engineer+ only.

Quando usar​

Caso 1: Falha de coleta​

Integração Sophos parou por 2h (credenciais expiradas).

Após fix, você quer recuperar os 2h de eventos perdidos.

Solução: Backfill de 2h atrás.

Caso 2: Novo vendor adicionado​

Você adicionou Defender ontem. Mas quer eventos dos últimos 7 dias.

Solução: Backfill de 7 dias.

Caso 3: Novo mapping criado​

Você criou regra que detecta novos padrões de ataque.

Quer reprocessar histórico 30 dias para "reatroativamente" detectar.

Solução: Backfill + reprocess.

Interface​

Acessar backfill​

Administração → Integrações → clique integração → Backfill.

Preencher formulário​

Data inicial:     2026-04-20 00:00:00 (de quando)
Data final:       2026-04-27 23:59:59 (até quando)
Throttle (req/s): 10 (velocidade máxima de request)

Data:

  • Passado máximo: depende de retenção vendor (Sophos guarda 90 dias típico).
  • Futuro: não permitido.

Throttle:

  • Baixo (1-5): lento, mas menos stress em vendor API.
  • Alto (10-50): mais rápido, risco de 429 rate limit.
  • Padrão: 10 (balanceado).

Executar​

Clique Iniciar backfill.

Status: "Backfill em progresso".

Você pode:

  • Fechar a página (continua em background).
  • Voltar para monitorar progresso.
  • Pausar via botão (pausa imediatamente, sem descartar coleta).

O que acontece​

1. CentralOps enfileira N tasks de coleta com data_range = [início, fim]
   └─ Divididas em chunks (ex: 1 dia por task, para não sobrecarregar)

2. Collector workers executam tarefas
   └─ Sophos API: GET /alerts?from=2026-04-20&to=2026-04-21
   └─ Respeita throttle: máx 10 req/s

3. Eventos voltam para normalização
   └─ Mapeamento DSL aplica (mapping atual)
   └─ Se erro: quarentena

4. Eventos normalizados são enviados ao Wazuh
   └─ Datatime preservado (não "now", mas original)

5. Pipeline Health mostra "Backfill: 65% concluído"
   └─ Você acompanha progresso

6. Concluído: "Backfill finalizado, 5000 eventos importados"

Performance​

Exemplo: Backfill 7 dias Sophos​

  • Eventos típicos: ~100/dia.
  • Total: ~700 eventos.
  • Requests: ~70 chamadas API (chunks de 10 eventos).
  • Duração: ~7 minutos (throttle=10).

Fatores​

  • Tamanho do range: 7 dias = 7x mais tempo que 1 dia.
  • Throttle: throttle=1 = 10x mais lento que throttle=10.
  • Vendor API: alguns vendors são lentos (dependência).

Monitoramento​

Dashboard​

Pipeline Health mostra:

  • Status: "Backfill running"
  • Progresso: "2500/5000 eventos (50%)"
  • Taxa: "15 eventos/min"
  • ETA: "~3 minutos para concluir"

Logs​

docker compose logs collector-worker-priority | grep backfill

Exemplo:

2026-04-27 10:00:01 [INFO] Backfill task started: integration_id=42, range=[2026-04-20, 2026-04-21]
2026-04-27 10:00:05 [INFO] Collected 100 events, cursor at 2026-04-20 12:30
2026-04-27 10:00:10 [INFO] Backfill task completed: 100 events imported

Pausar e retomar​

Clique Pausar backfill:

  • Workers param de coletar.
  • Cursor é salvo (próxima tarefa continua de onde parou).
  • Você pode retomar depois.
Exemplo:
  Pausa às 10:00 (2500/5000)
  Retoma às 14:00
  Continua de onde parou (mesma taxa)

Reprocessar com novo mapping​

Workflow: Backfill → novo mapping → reprocess.

1. Backfill 30 dias de Sophos (eventos em quarentena ou normalizados com v1)
2. Você cria novo mapping v2 (detecta novos padrões)
3. Vá a Quarentena → "Reprocess all Sophos" (se houve erros)
4. Ou: eventos normalizados com v1 não são reprocessados automaticamente
   → Requery via API (roadmap)

Throttle: quando ajustar?​

Aumentar throttle (mais rápido)​

Se:

  • Vendor não reclama (sem 429).
  • Você tem urgência (quer dados hoje).
De: 10 → Para: 20
Resultado: ~2x mais rápido
Risco: possível 429, backoff automático

Diminuir throttle (mais conservador)​

Se:

  • Vendor está 429ing você (rate limit).
  • Vendor está sobrecarregado (performance ruim).
De: 10 → Para: 5
Resultado: ~2x mais lento
Benefício: menor strain no vendor

Troubleshooting​

"Backfill parou no meio"​

Causa: worker crash, vendor retornou erro, throttle muito alto
Log: docker compose logs collector-worker-priority | grep error
Ação: verificar erro, pausar, retomar quando corrigido

"Taxa muito lenta"​

Causa: throttle baixo demais OU vendor API lenta
Ação:
  1. Aumentar throttle em 5
  2. Se ainda lento: pode ser vendor (espere ou contate suporte)

"429 Too Many Requests"​

Causa: throttle muito alto
Ação:
  1. Pausar backfill
  2. Diminuir throttle (ex: 10 → 5)
  3. Retomar
  4. Celery faz backoff automático (retry)

Casos de uso avançados​

Multi-vendor backfill​

Backfill Sophos 7 dias + Defender 7 dias em paralelo:

1. Abra integração Sophos → inicie backfill
2. Abra integração Defender → inicie backfill
3. Ambas rodam em paralelo (filas separadas)

Backfill + mapeamento customizado​

1. Backfill 30 dias de Sophos (com mapping v1)
2. Você cria mapping v2 com novos campos
3. Imediatamente, novos eventos usam v2
4. Antigos continuam com v1 (não reprocessados)
5. Para reprocessar todos: (roadmap) ou manualmente descartar + recoleta
