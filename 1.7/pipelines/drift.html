Drift Detection

Sistema automático de descoberta de campos não-mapeados.

Público: Operator+.

O que é drift?​

Quando um evento bruto chega com campo que não está no mapping DSL, ele vai para drift.

Exemplo:

Evento Sophos:
{
  "id": "alert-123",
  "severity": 5,
  "title": "Malware",
  "ai_score_numeric": 0.95        ← Campo novo! Não está no mapping
}

Mapping:
  timestamp = ...
  severity = ...
  title = ...
  # ai_score_numeric não aparece

Resultado: ai_score_numeric entra em drift

Por quê isso importa?​

Campos em drift são invisíveis no evento normalizado. Podem ser importantes:

  • ai_score_numeric → confiança da detecção (útil para filtragem).
  • threat_family → tipo de malware (contexto).
  • remediation_status → se foi deletado (ação tomada).

Se ignorados, você perde contexto.

Interface​

Vá a Operação → Drift Explorer (ou Dashboard → card Drift).

Tabela com:

ColunaDescrição
CampoNome do field (ex: ai_score_numeric).
VendorPlataforma de origem (sophos, defender).
TypeTipo inferido (string, number, boolean).
CountQuantas vezes visto (últimas 24h).
Last seenTimestamp do último evento com esse field.
Statusunknown, ignored, marked_mapped.
AçãoIgnore / Mark as mapped.

Workflow​

Cenário: campo novo apareceu​

  1. Sophos retorna novo campo ai_score_numeric.
  2. Drift detector identifica (automático).
  3. Você vê em Drift Explorer.

Decisão:

Opção A: Ignorar​

"Este campo não é importante. Pare de mostrar."

Clique: Ignore
Status: muda para "ignored"
Resultado: não aparece mais em drift (mas continua nos dados brutos se quiser examinar)

Use quando: campo é realmente irrelevante (ex: timestamp interno do vendor).

Opção B: Marcar como mapeado​

"Já tratei isso. Deixa de alerta."

Pré-requisito: você editou o mapping DSL para incluir o field.

Exemplo:
  1. Vê campo "ai_score_numeric" em drift
  2. Vai a Mappings → edita mapping Sophos
  3. Adiciona: ai_score = ai_score_numeric
  4. Salva (v2)
  5. Volta a Drift Explorer
  6. Clique "Mark as mapped"
  7. Status: marked_mapped

Resultado: campo desaparece de drift (você o tratou).

Opção C: Criar novo field no mapping​

"Este campo é importante. Quero incluir em eventos normalizados."

Processo:

  1. Veja drift → campo ai_score_numeric.
  2. Vá a Mappings → DSL do vendor.
  3. Adicione regra: confidence_score = float(ai_score_numeric).
  4. Salve (cria versão v2).
  5. Volte a Drift → Mark as mapped.
  6. Eventos novos incluem confidence_score.
  7. Repositorio antigos em quarentena → Reprocess com novo mapping.

Filtros​

Por vendor​

Dropdown: Sophos, Defender, NinjaOne, etc.

Útil: "Quais campos novos Sophos está mandando?"

Por status​

  • unknown — visto mas não acionado.
  • ignored — você decidiu ignorar.
  • marked_mapped — você incluiu no mapping.

Por tipo​

  • string — texto.
  • number — int ou float.
  • boolean — true/false.
  • array — lista.
  • object — JSON aninhado.

Bulk actions​

Se muitos campos desconhecidos:

Filtro: Vendor = Sophos, Status = unknown
Resultado: 50 campos
Clique: "Ignore all"
Status: todos marcados como ignored

Ou:

Clique: "Mark all as mapped" (se você já editou mapping em lote)

Casos de uso​

Vendor atualizou schema​

Sophos adicionou novos campos em sua API.

  1. Vá a Drift.
  2. Filtro Vendor = Sophos.
  3. Vê 10 campos novos.
  4. Anota quais são importantes (ex: ai_score_numeric).
  5. Edita mapping Sophos, adiciona esses campos.
  6. Mark as mapped.
  7. Reprocess quarentena se houver eventos com erro.

Investigação: "Que dados estou perdendo?"​

  1. Abra Drift.
  2. Ordena por "Count" (mais frequentes).
  3. Top 10 campos são os mais importantes (se não estão mapeados).
  4. Decisão: ignorar ou incluir?

Auditoria de completude​

Compliance: "Estou coletando todos os campos relevantes?"

  1. Drift Explorer → filtro ignored.
  2. Revisa: realmente irrelevantes? Ou falta mapping?
  3. Se falta mapping: marque para próximo sprint.

Limpeza​

Drift é retenção 90 dias (padrão, configurável por org).

Após 90 dias sem ver campo, é automaticamente deletado.

Se campo reaparece depois, vai para drift novamente (fresh).

Performance​

  • Overhead: mínimo. Detecção é side-effect de parsing.
  • Storage: drift entries são pequenas (~100 bytes cada).
  • Escala: testado com 1000+ campos por vendor.
