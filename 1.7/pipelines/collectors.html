Collectors (Celery workers)

Workers Celery que puxam eventos das APIs de vendors de forma paralela e escalável.

Público: Engineer+. Admin vê health.

Arquitetura​

┌─────────────────────────────────────────────────────┐
│ Celery Beat (scheduler singleton)                   │
│  ├─ RedBeat armazena schedule no Redis              │
│  └─ Dispara tasks a cada 1–5 min                    │
└─────────────────────┬───────────────────────────────┘
                      │
      ┌───────────────┴───────────────┐
      ▼                               ▼
┌──────────────────┐      ┌──────────────────┐
│ collect.priority │      │ collect.bulk     │
│ FIFO queue       │      │ FIFO queue       │
└────────┬─────────┘      └────────┬─────────┘
         │                         │
    ┌────┴──────────────┬──────────┴────┐
    ▼                   ▼               ▼
  Worker-1           Worker-3        Worker-5
  Worker-2           Worker-4        Worker-6
  (N replicas)       (M replicas)    (batch)
    │                   │               │
    ├─ Sophos alerts    ├─ NinjaOne     └─ Batch cleanup
    ├─ Defender        │   activities    tasks
    └─ Real-time       └─ Audit logs

    ▼▼▼ (eventos coletados)
    Redis (dedup, cursor)
    PostgreSQL (RawEvent)
    Mapeamento DSL
    ▼▼▼ (eventos normalizados ou quarentena)

Componentes​

Beat scheduler​

  • Tipo: Celery Beat com RedBeat backend.
  • Escala: 1 singleton (não replicado).
  • Função: ler schedule do Redis, disparar tasks no tempo certo.
  • Redeploy: se morrer, relança automaticamente via Docker healthcheck.

Priority queue (collect.priority)​

Integrações de tempo real (alertas, detecções):

  • Sophos alerts (1 min).
  • Defender incidents (2 min).

Workers: N réplicas (padrão 2, escalar conforme load).

Performance: < 1s latência do evento ser criado até estar em fila.

Bulk queue (collect.bulk)​

Integrações de auditoria/histórico:

  • NinjaOne activities (5 min).
  • Audit logs (nightly).

Workers: M réplicas (padrão 1, baixa prioridade).

Performance: OK latência de minutos.

Escalação​

Aumentar workers on-the-fly​

# Aumentar priority workers para 4
docker compose up -d --scale collector-worker-priority=4

# Aumentar bulk workers para 2
docker compose up -d --scale collector-worker-bulk=2

Novos workers pegam tarefas da fila em < 1s.

Diminuir workers​

# Voltar priority para 2
docker compose up -d --scale collector-worker-priority=2

Workers em execução terminam gracefully, novos não são iniciados.

Monitoramento de fila​

Vá a Pipeline Health → veja Queue depth para cada fila.

Se collect.priority depth > 100 e crescendo:

  • Adicionar workers.
  • Ou otimizar cada collector (ex: cursor mais eficiente).

Concorrência​

Tipo: prefork (não thread).

Cada worker replica é processo separado, não thread.

Vantagem: sem GIL (Global Interpreter Lock), paralelismo real.

Desvantagem: mais memória por worker (~50-100MB base).

Token refresh​

Collectors fazem refresh automático de tokens OAuth:

1. Collector precisa de token (ex: Sophos access_token)
2. Verifica Redis: oauth:token:integration_id
3. Se válido: usa
4. Se expirado: refresh token (Refresh Token Flow)
5. Armazena novo token em Redis com TTL
6. Log: "Token refreshed for integration_id"

Lock Redis previne race condition (múltiplos workers tentam refresh ao mesmo tempo).

Cursor persistência​

Cada collector mantém cursor no Redis:

Key: collection:cursor:integration_id:stream
Value: {"timestamp": "2026-04-27T10:00:00Z", "id": "last_event_id"}
TTL: 7 dias (retenção)

Delta sync: próxima coleta começa de cursor.timestamp, não desde zero.

Idempotência: mesmo evento coletado 2x é dedupado via message_id.

Dedup​

Redis Set com TTL:

Key: dedupe:integration_id
Member: vendor_message_id
TTL: 24h

Se evento chegaem 2x em 24h: segunda cópia é descartada (log: "Duplicate event skipped").

Error handling​

Se collector falha:

1. Exceção capturada
2. Log do erro (stack trace)
3. Task marcada como "failed" (não retried automaticamente)
4. Event continua fila para próxima coleta
5. Alert em Pipeline Health: "Integration error"

Se erro é transiente (429 rate limit, timeout):

Celery retorna em 1 min (exponential backoff)

Se erro é permanente (401 unauthorized, invalid API key):

Integração vai para estado "error"
Health indicator vira vermelho
Admin é alertado

Logging​

Logs aparecem em docker compose logs collector-worker-priority:

2026-04-27 10:00:01 [INFO] collect_vendor_logs_priority[integration_id=42, stream=alerts]
2026-04-27 10:00:02 [INFO] Collected 23 events from sophos
2026-04-27 10:00:03 [INFO] Token refreshed for integration_id=42
2026-04-27 10:00:05 [INFO] task completed successfully

Erro exemplo:

2026-04-27 10:05:01 [ERROR] collect_vendor_logs_priority failed: 401 Unauthorized
2026-04-27 10:05:01 [ERROR] integration_id=42: Credentials invalid or expired
2026-04-27 10:05:01 [ERROR] Retrying in 60 seconds...

Performance tuning​

Slow collector?​

Profiler:

# Dentro worker, adicionar
import py_spy
py_spy.start_sampling()  # via env var

Causa comum: API lenta (grande volume de eventos).

Solução: aumentar intervalo de coleta (ex: 2 min em vez de 1 min), descartando eventos antigos via cursor.

High memory?​

# Verificar memory por worker
docker stats collector-worker-priority

Se > 200MB, pode ser leak em loop de coleta.

Solução: revisar vendor API response (está retornando muitos events?), aumentar batch size em config.

Queue growth?​

Se queue depth cresce continuamente:

Rate de entrada > Rate de saída
Solução: escale workers ou otimize cada evento

Shutdown graceful​

Quando você restarta:

docker compose down

Beat + workers terminam gracefully:

  • Tasks em execução completam.
  • Cursores são salvos.
  • Tokens refreshados são preservados.

Sem perda de dados.
